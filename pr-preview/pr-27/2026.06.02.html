    発表者: Ali Caglayan 概要: Dune package management は、dune に opam パッケージのビルドおよび使用方法を習得させることを目的としたプロジェクトです。2023年から活発に開発が進められており、より洗練された有用なものへと形を整えてきました。現時点ではまだ実験的と位置付けられていますが、本発表では現在できること（再配置可能なコンパイラ、lockless ビルド、OxCaml サポート）を紹介し、現在開発中の機能（in-and-out、dev-tools）についても取り上げ、安定版へのロードマップの展望をお伝えします。

Slipshow！

Archive: https://discuss.ocaml.org/t/ann-slipshow/16337/23

このスレッドを引き継いで、Paul-Elliot が発表しました

まず小さなお知らせです：Slipshow に専用の Zulip ができました！プレゼンの準備中にサポートを受けたり、力作を披露したりできます！

すべての手続きを終えた今、opam での slipshow の次のリリースを正式にお知らせします：

Slipshow 0.11.0: Brazlip

タイポが 劇的な 結果をもたらすことはご存知のことでしょう。そのため、私たちはタイポを何としても防ぎたいと考えています。

このリリースにより、slipshow はエラーをエディタ上に直接表示できるようになりました。slipshow はあなたが書くすべてを監視し、あらゆる警告、タイポ、その他の問題のある動作を即座に報告します。

以下は、プレゼンテーション中のタイポがライブで修正される様子の録画です：

https://github.com/user-attachments/assets/ac675cdf-1d08-41e4-b384-25a3bfd764ef

LSP サーバーがサポートする機能：

  • 開いているバッファへのエラー表示
  • ID を引数として使うアクションにおける ID の補完
  • アクションのホバー時にドキュメントを表示
  • ID の出現箇所のハイライト
  • 特定の ID の定義へのジャンプ
  • マルチファイルプレゼンテーション
  • 同期プレビューサーバー
  • 保存時のコンパイル

リリースのたびに、素晴らしい OCaml ライブラリを発見することに驚かされます。今回は @c-cbue による linlol です。作成・メンテナンスしてくださり、本当にありがとうございます！

いつものように、NNLet の惜しみない貴重なサポートにも感謝いたします。そして…❤️ 私のスポンサーの皆様にも！ 

$ opam update
$ opam upgrade slipshow

このリリースの改善点は LSP サーバーだけではありません。全変更履歴もご覧ください！

追加

  • プレビュアーの状態（切断中、更新中など）を示すビジュアルインジケーターを追加 (#220, #222)
  • インクルードされたファイルでのフロントマターのサポートを追加 (#228)
  • (#229) エディタと連携するための LSP サーバーを追加（slipshow lsp コマンドで利用可能）。サーバーがサポートする機能：

doctor 0.3.0: 読み取り専用 OCaml 環境診断ツール

Archive: https://discuss.ocaml.org/t/ann-doctor-0-3-0-read-only-ocaml-environment-diagnostics/18186/1

Funwithcthulhu が発表しました

こんにちは、

doctor 0.3.0 をリリースしました。

doctor は、ローカルの OCaml 開発環境をチェックするための小さな読み取り専用 CLI ツールです。opam、スイッチ、PATH、dune、ocaml-lsp-server、ocamlformat、VS Code OCaml Platform 拡張機能に関するよくある設定の問題を報告します。

インストール： 

opam update
opam install doctor

基本的な使い方： 

doctor check
doctor check --json

0.3.0 リリースでは、opam スイッチ環境の診断に重点を置いています。特に、ツールが見つからない場合と、opam にアクティブなスイッチはあるが現在のシェルがそのスイッチ環境を読み込んでいない場合を区別しようとしています。

例えば、ocaml がアクティブなスイッチ外で解決されている場合や、dune、ocamllsp、ocamlformat などのインストール済みのスイッチツールが PATH に存在しない場合を報告できます。

このコマンドは opam init の実行、スイッチの作成、パッケージのインストール、シェル/エディタファイルの編集を行いません。確認した内容を報告し、手動で実行するコマンドを提案するのみです。

JSON 出力は、安定した診断名と終了コードのマッピングを必要とするエディタ統合や小さなスクリプト向けに設計されています。

リポジトリ：

https://github.com/funwithcthulhu/doctor

opam パッケージ：

https://opam.ocaml.org/packages/doctor/

わかりにくい診断や、よくある設定ケースの欠落に関するフィードバックをいただけると助かります。

ありがとうございます！

Solo5: unikernel 向けサンドボックス実行環境

Archive: https://discuss.ocaml.org/t/ann-solo5-a-sandboxed-execution-environment-for-unikernels/18191/1

Calascibetta Romain が発表しました

Solo5 0.11.0 のリリースを喜んでお知らせします。このリリースは、OCaml における unikernel のパフォーマンス改善の始まりを示すものとして重要な意味を持ちます。

プロジェクト自体は C で書かれていますが、unikernel 開発の礎となっています。OCaml サポートは ocaml-solo5（OCaml コンパイラのバージョンを追従）を通じて提供されています。以下は OCaml を使った Solo5 (hvt) unikernel の簡単な例です： 

$ opam install ocaml-solo5
$ eval $(opam env)
$ cat >startup.c<<EOF
#include "solo5.h"
#include <caml/callback.h>

static char *argv[] = { "unikernel", NULL };

void _nolibc_init(uintptr_t, size_t);

int solo5_app_main(const struct solo5_start_info *si) {
  _nolibc_init(si->heap_start, si->heap_size);
  caml_startup(argv);
  return (0);
}
EOF
$ cat >manifest.json<<EOF
{
  "type": "solo5.manifest",
  "version": 1,
  "devices": []
}
EOF
$ cat >main.ml<<EOF
let () = print_endline "Hello World!"
EOF
$ solo5-elftool gen-manifest manifest.json manifest.c
$ ocamlfind -toolchain solo5 opt \
  manifest.c startup.c main.ml \
  -cclib "-z solo5-abi=hvt" -o main.hvt
$ solo5-hvt -- main.hvt
            |      ___|
  __|  _ \  |  _ \ __ \
\__ \ (   | | (   |  ) |
____/\___/ _|\___/____/
Solo5: Bindings version v0.11.0

OCaml で unikernel を開発することについてさらに詳しく知りたい方は、（直近の MirageOS リトリートで使用した）スライドや、こちらで公開している mnet のチュートリアルをご覧ください。

Solo5 とは？

Solo5 は、サンドボックス内でプログラムを作成・実行するための C フレームワークとツール群です。このサンドボックスは当初、外部世界との相互作用を次の5つのみに限定しています：

  1. net デバイスへの Ethernet フレームの読み書き
  2. block デバイスへのページの読み書き
  3. 終了

サンドボックス環境で利用可能なすべての関数はこのヘッダに記述されています。

Solo5 の注目すべき点は、この API が VirtIO、Xen（および QubesOS）、そして私たちの最小限のハイパーバイザ hvt（KVM、BHyve、VMM 上で動作）と連携できることです。これにより、Solo5 アプリケーションは幅広いプラットフォームにデプロイできます。Ubuntu では、Solo5 を次のコマンドでインストールできます： 

$ curl -fsSL https://apt.robur.coop/gpg.pub | \
  gpg --dearmor > /etc/apt/trusted.gpg.d/apt.robur.coop.gpg
$ echo "deb [signed-by=/etc/apt/trusted.gpg.d/apt.robur.coop.gpg] https://apt.robur.coop ubuntu-24.04 main" > /etc/apt/sources.list.d/robur.list
$ apt update
$ apt install solo5

hvt unikernel

Robur では hvt unikernel の構築を好んでいます。これらの unikernel は実行に solo5-hvt "tender"（約200KB）のみを必要とし、Linux、FreeBSD、OpenBSD で利用可能です（WSL は ネストされた 仮想化をサポートしているため、WSL 上でも unikernel を実行できます！）。

hvt の利点はその最小主義にあります（Solo5 全般の特徴）。この最新リリースでは、libseccomp を使用して Linux 上のサンドボックスが完成し、unikernel の実行に必要な権限のみを保持するようになりました（合計18のシステムコール）。このサンドボックス機能は、FreeBSD では Capsicum、OpenBSD では pledge(2) によってすでに利用可能でした。

このリリースでは、eventfd(2) と tap インターフェースに直接書き込む並列スレッドを活用することで、Linux 上の tap インターフェースへの Ethernet フレーム書き込みスループットも改善されています。これにより、コストのかかる VM 退出の回数を減らすことができます。

この目的のため、iperf3（および mnet を使ってこのプロトコルを実装した unikernel）を使ってベンチマークを実施し、スループットの改善を実証しました。

solo5.0.10.1 solo5.0.11.0
~700 Mbits/sec ~2.15 Gbits/sec

unikernel のデプロイ

Robur では、unikernel のデプロイをできる限り簡単にすることを目指しています。そのため、さまざまなサービスをデプロイするために設計されたソフトウェアと unikernel のスイートを開発しました：

  1. Albatross は（apt でもインストール可能な）プロジェクトで、hvt unikernel の起動と監視が可能なデーモンを実行できます。このサービスは hvt unikernel 専用に設計されており（最近ではシステムのデプロイも可能になりましたが）、利点があります。
  2. dnsvizor は IP アドレッシングと DNS 解決の問題を解消する unikernel です。このような unikernel を起動し、以降の unikernel がそれを基準にネットワークを設定するという考え方です。
  3. 最後に、mollymawk を積極的に開発中です。これは新しい unikernel を（できるだけシンプルに）デプロイするための Albatross 向け Web インターフェースを提供する unikernel です。

また、aussi（まだ実験的な段階）も開発しました。これは unikernel 向けの OCI ランタイムです。これにより、Docker を使って unikernel をビルド・デプロイすることも可能です。unikernel annuaire を使った例がこちらで公開されています。

unikernel の開発

最後に、unikernel 開発のもう一つの側面があります。Robur では、潜在的な新規ユーザーが新しい unikernel を自ら開発できるよう（所有感を持てるよう）、ライブラリ（特にプロトコルとフォーマット）の開発にも取り組んでいます。

ぜひ、このテーマについてご質問ください！

この取り組みの一環として、私たちは標準的な unikernel の開発を可能にする多数のライブラリを開発してきました：

  • unikernel（およびサービス）のスケジューラとして Miou の開発を継続しています
  • unikernel 向けの新しい TCP/IP スタックである mnet と utcp を開発しています
  • unikernel 向け FAT32 ファイルシステムである mfat も利用可能です
  • Web サイトを unikernel として開発するための Web フレームワーク vifu も利用可能です
  • そして、長年にわたってメンテナンスしてきた多くの他のプロジェクトも！

まとめ

このリリースは、私たちの協同組合の活動の包括的（かつアクセスしやすい）な概要をお伝えする良い機会となりました。この活動は本質的に、OCaml での unikernel の開発とデプロイに必要なコンポーネントを作成しながら、コミュニケーションと生産手段の主権回復という理念を推進することを意味しています。

Happy hacking!

皆さん、こんにちは！今年の OCaml Workshop はパリで開催されます。ワークショップへの発表提案を募集いたします。

重要情報

  • 開催地: パリ（FPW 2026 と併催）
  • 日程: 8月27日（変更の可能性あり！ 代替候補: 8月31日）
  • 提案締め切り: 7月1日（AoE）
  • 採択通知: 7月27日

スコープ

OCaml Workshop は、OCaml の魅力を発見中の愛好家から、unsafe な魔法を使いこなすウィザードまで、幅広い OCaml ユーザーを歓迎します。共通点は OCaml への情熱と、もっと学びたい・同じ OCaml ユーザーとつながりたい・言語を改善する方法を一緒に見つけたいという想いです。

トーク提案も同様に幅広く歓迎します：OCaml に関連するものであれば何でも歓迎です！

以下のトピックに限定されるわけではありませんが、過去の例としては：OCaml 編集ツール、検証済み OCaml 成果物、OCaml と他言語との相互運用性、OCaml の行動規範、コンパイラ最適化、OS ポータビリティ、OCaml テストフレームワーク、OCaml での並行処理パッケージなどがあります。

過去のエディションの全カタログは以下のリンクからご覧いただけます：

  • 2025: https://ocaml.org/conferences/ocaml-workshop-2025
  • 2024: https://ocaml.org/conferences/ocaml-workshop-2024
  • 2023: https://ocaml.org/conferences/ocaml-workshop-2023
  • 2022: https://ocaml.org/conferences/ocaml-workshop-2022
  • 2021: https://ocaml.org/conferences/ocaml-workshop-2021

形式

標準トーク（20分）に加えて、以下の形式も受け付けています：

  • デモ. 30分のチュートリアル形式によるツールのデモンストレーション。
  • 見解の表明. OCaml の設計空間（機能の追加や削除など、これに限らず）に関するトピックについての20分のプレゼンテーション。
  • 経験報告. OCaml またはツールの使用に関する20分の報告。

    • OCaml コミュニティへの関連性

LLM 使用に関する推奨事項

LLM によって大部分が書かれた提案は受け付けられず、デスクリジェクトされます。文法の修正やスタイルの向上に LLM を使用することは全く問題ありません（特に英語が母語でない場合）が、コンテンツを直接生成するために使用することは危険であり非専門的であり、著者性とレビュアーの労力を損ないます。
