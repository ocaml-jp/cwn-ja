http-date v0.2 — OCaml向けゼロ依存HTTPデータ解析

Archive: https://discuss.ocaml.org/t/ann-http-date-v0-2-zero-dependency-http-date-parsing-for-ocaml/17984/1

Bikal Lem が発表しました

RFC 9110 準拠の OCaml 向け HTTP 日時デコーダー/エンコーダー、http-date の v0.2 をリリースしました。

v0.1 からの大幅な書き直しです。主な変更点は以下の通りです：

ゼロ依存 — ocamllex/menhir パーサーを手書き実装に置き換え、ptime への依存を完全に除去しました。このライブラリは現在、ocaml と dune のみを必要とします。

新しい型安全な API — 解析結果はフォーマット（IMF、RFC850、`ASCTIME）でタグ付けされた多相バリアントとなり、どの HTTP 日付フォーマットが解析されたかを常に把握できます。dayname、date、time、datetime 型はすべてパブリック API で公開されています。

本格的なテスト — alcobar によるプロパティベーステストと AFL ファジングインフラを追加し、手書きパーサーへの信頼性を高めました。

OCaml で HTTP ヘッダーを扱っている方はぜひお試しください：

opam install http-date

https://github.com/bikallem/http-date

Caqti 2.3.0

Archive: https://discuss.ocaml.org/t/ann-caqti-2-3-0/17985/1

Petter A. Urkedal が発表しました

Caqti 2.3.0 のリリースをお知らせします。

TL;DR: リンク行に caqti.classic を追加することを検討してください（このバージョンから利用可能）。現時点では caqti ライブラリを再エクスポートするだけですが、Caqti 3 では段階的な移行を可能にするために Caqti 2 API の互換レイヤーに置き換えられる予定です。

リリースノート

このリリースの新機能の一部は、まだ不安定な caqti.template ライブラリを通じてのみアクセス可能であることに注意してください。このライブラリは、近日公開予定の Caqti 3 API の一部として間もなく安定化される予定です。モジュール名の調整により、caqti.template への移行は将来の互換性を保証しません。代わりに、Caqti 3 への移行を容易にするために、新しい caqti.classic を依存関係に追加することを推奨します。

  • caqti ライブラリのエイリアスとして caqti.classic を追加しました。Caqti 3 がリリースされると、現在の Caqti 2 API との互換性を提供するライブラリになります。段階的な移行を可能にするために、2 つの API を並行して使用できます。
  • caqti.template における MariaDB のダイアレクト記述子がサーバーバージョンを提供するようになりました。
  • caqti.template ライブラリに Query.parens、Query.litf、Query.vars、Row_type.fields を追加しました。
  • パラメトリック型を適用的にインスタンス化できるよう、行型作成のインターフェースを改訂しました。この変更以前は、パラメトリック型を表す関数の各適用は、同じ型パラメータ引数に適用された場合でも新しい型 ID を生成していました。このインターフェースの中心となるのは Caqti_template.Constructor_type モジュールです。
  • リクエストテンプレートに提供されるクエリ関数はメモ化されるようになり、ダイアレクトごとに 1 回だけ呼び出されます。これは最適化を目的としたものですが、ここで副作用を避けることは依然として良い習慣です。
  • 新しい関数 Query.with_pos_of により、クエリジェネレーターが構文ツリーにソース位置を追加できるようになりました。enable_query_annotations が設定されている場合、これはクエリ文字列内の SQL コメントとして出力されます。クエリがアプリケーションコードの異なる部分から組み立てられる場合のデバッグが容易になります。
  • まだ不安定な caqti.template ライブラリが、単一のリクエストテンプレートで複数のステートメントをサポートするようになりました。許可される場合は、単一のクエリとして日和見的に送信されます。これはまだ実験的です。
  • sqlite3 のシムルールのパッケージスコープを修正しました（#133、mefyl による）。
  • SQLite3 ドライバーのエラー分類を修正しました（#132）。
  • PostgreSQL の接続損失後の再接続を修正しました。

ptt：unikernel としてのメーリングリストシステム

Archive: https://discuss.ocaml.org/t/ann-ptt-a-mailing-list-system-as-unikernel/17987/1

Calascibetta Romain が発表しました

OCaml unikernel だけで動く新しいメーリングリストを立ち上げました。ウェブサイト（それ自体も unikernel）は https://mailingl.st です。SMTP 関連の unikernel の開発とデプロイに興味がある方は、ptt-subscribe@mailingl.st にメールを送ることで ptt@mailingl.st を購読できます。

注意：現時点ではまだ公開テストメーリングリストです。長期的には、ptt プロジェクトに焦点を当てる予定です。

SMTP プロトコル：長く曲がりくねった道！

  • 始まりは電子メール

    すべては Mr.MIME、つまりメールのデコードとエンコードのためのライブラリから始まりました。関連する RFC を体系化したものですが、より重要なのは、IEEE、Enron、KVM、そして最近では caml-list の実際のメールでバトルテストが行われていることです。

    この作業により、ファジングツールを使用して生成された有効なランダムメールのデータベース Hamlet も構築できました。

    内部では、Mr.MIME は（rosetta を介した国際化サポートを持つ）メール内で見られる値の最も一般的な形式をデコードするための unstrctrd と、SMTP 制約と（コメント）折りたたみ空白処理を尊重しながらメールをエンコードするための prettym に依存しています。

  • 次にプロトコル

    次に、SMTP プロトコルの OCaml 実装である colombe が登場しました。STARTTLS サポートには ocaml-tls を使用しています。

    プロトコルは仮に「シンプル」（ただしインターネットは常にサプライズをもたらします）ですが、最初からスケジューラーやネットワーク層から独立して設計しました。そのため、unikernel に摩擦なく組み込めます。

  • 最後に、正当性

    これらのコアコンポーネントの上に、いくつかのメールセキュリティレイヤーを構築しました：

    • ocaml-dkim はストリーミング方式でのメール整合性の署名と検証を処理します（検証と署名生成の両方）
    • uspf は送信者 ID を検証し、ほとんどのライブラリと同様にスケジューラーや DNS 実装から独立しています
    • ocaml-dmarc は DKIM と SPF の検証を自動化し、結果でメールにスタンプを押し、ドメイン名全体のアライメントを確認します
    • [ocaml-arc](github.com/robur-coop/ocaml-arc) は、メールが複数の SMTP サーバーを通過する際（メーリングリストで起きることがまさにこれです）に信頼の連鎖を完成させるためにメールを検証・署名できます

    これらすべてについての短い記事をこちらに書きました。

すべては unikernel の形で

最初の実験ですでに MirageOS unikernel でメールを処理できることを示しましたが、メモリリーク、セキュリティの脆弱性、ビルドの問題という実際の制限にも直面しました。

そこで最初からやり直すことにし、OCaml 5 とエフェクトを完全に活用する機会としました。主要な部分を一から作り直しました：

  • 新しいエフェクトベースのスケジューラー：Miou/Mkernel
  • より完全な TCP/IP スタック：Mnet
  • 新しい FAT32 ファイルシステム：Mfat

ptt はこの新しいスタック上に構築されており、今のところメモリリークは観察されていません（メモリ使用量のトレースには mkernel-memtrace を、表示には memtrace-viewer を使用）。mirage-tcpip に関する CVE は mnet の開発時に考慮され、ビルドの仕組みも大幅にシンプルになりました。GitHub アクションが unikernel をビルドして実際に実行してテストできます。mnet で確認できます。

このアプローチを使用した他の unikernel も利用できます。興味があれば、OCaml で unikernel を作成するためのこのチュートリアルをご覧ください。

デプロイ

ptt はデプロイの問題にも取り組んでいます。ptt の「ステートレス」な側面を紹介する記事があります。また、セキュアな unikernel オーケストレーター [Albatross](github.com/robur-coop/albatross) と、unikernel をデプロイするためのウェブインターフェース（それ自体も unikernel）Mollymawk を（再）紹介したいと思います。

より広く言えば、これが私たちの協同組合が取り組んでいることです：開発者でもデプロイヤーでも、ユーザー体験を向上させたいのです。私たちは、unikernel を実際に開発、デプロイ、使用することが、より広く採用されるための唯一の方法だと信じています。これらのプロジェクトもぜひフォローしてください！

使用法

この過程で、メールのライフサイクルのすべての段階を追跡できるツールが非常に役立つことがわかりました。そこから blaze が生まれました：メール処理のためのスイスアーミーナイフです。

まだ実験的ですが、すでに以下が可能です：

  • アーカイブシステムの使用（生成、読み込み、インデックス化など）
  • mbox や maildir などの他のアーカイブの処理
  • POP3 プロトコルを介した通信
  • メールの署名と検証（DKIM と ARC）
  • コマンドラインからのメール作成
  • メールの送信
  • 小さなローカル SMTP サーバーの実行

blaze はライブラリ API を反復し実装を検証する方法です。実験的ですが、徐々に完全なメールクライアントに成長しています。

  • アーカイブとインデックス化

    stem プロジェクトもご紹介したいと思います。これはドキュメント（メールなど）から単語の語根を抽出してトークン化し、自然言語の複雑さなしに分析できるものにします。このトークン化が私たちの小さな bm25 検索エンジンを動かしています。結果はこちらで確認できます。

    これはまた、unikernel として利用可能な私たちの caml-list 検索エンジン blame を動かしているものでもあります。https://caml-list.robur.coop でお試しいただけます（vif によって提供）。

    検索の他に、Message-ID によるメールのインデックス化もあります。そのために bancos を構築しました：OCaml の並列アクセスをサポートする永続的な radix ツリーです！詳細はこちらです。

    最後に、インデックス化システムは（Git がオブジェクトを保存するのに使用するのと同じ）PACKv2 フォーマットを使用します。これは carton ライブラリによって実装されています。ocaml-git プロジェクトを通じてその安定性が証明されているため、メールのアーカイブに再利用することにしました（public-inbox が行ったこととは異なる形ですが）。

結論

この作業のおかげで、OCaml にはメール関連プロジェクトの強固なセットが揃いました。この旅は 2016 年に始まり、堅牢でバトルテストされたソリューションを提供することを常に目指しているため、まだ長い道のりがあります。他の言語での一部の実装とは異なり（Rust コミュニティの方々とも議論中ですが）、私たちのものは実際に標準に準拠しています！

大したことではないように見えるかもしれませんし、メールを単に交換するだけでは大きな違いはわからないかもしれませんが、このアプローチはより良いインターネットへの道を開くと信じています。unikernel の形で、それはコミュニケーション手段の真の取り戻しを表しています！

Mollymawk と Albatross によるすべての仮想マシンのオーケストレーション

Archive: https://discuss.ocaml.org/t/ann-mollymawk-and-albatross-orchestrating-all-virtual-machines/17988/1

Hannes Mehnert が発表しました

皆さん、こんにちは。

MirageOS unikernel でない仮想マシンもサポートできるよう、Albatross と Mollymawk を完成させてデプロイしました。

理由はシンプルです：メトリクス、コンソール出力、ウェブ UI からのデプロイ、マルチテナント、起動依存関係、障害時の再起動など、多くの優れた機能を組み込んだため、MirageOS でない仮想マシンも同じ仕組みでデプロイし、コードの重複を避けたかったのです。

現時点では FreeBSD BHyve のみをサポートしていますが、他の仮想化技術に興味がある方はお知らせください。優先的に対応します！

スクリーンショット付きの簡単なブログ記事も書きました：https://blog.robur.coop/articles/mollymawk-other.html

チューリングマシンが GADT に出会うとき — Florian Angeletti

GADT パターンマッチングで明示的な到達不能なケースを書く必要があるのはなぜか、疑問に思ったことはありますか？あるいは、OCaml 型の中にどれだけの計算を忍ばせることができるか、考えたことはありますか？

このトークでは、これらの質問などに答えるために、GADT、パターンマッチの網羅性チェックの OCaml コンパイラ実装、そして型チェッカーを騙して BB(3) チャンピオンを自力で見つけさせる最善の方法を深く掘り下げます。

OCaml のパターンマッチングの拡張 — Yanni Lefki

パターンマッチングは数十年にわたって研究され、広範な研究と多数の拡張の対象となっています。それにもかかわらず、Rust の if-let 構文などの最近の言語機能や、Cheng と Parreaux（OOPSLA 2024）などの最近の研究は、まだ改善の余地があることを示唆しています。私たちはパターンマッチングを条件分岐の拡張形式と統一するスリム化されたアプローチを提案します。

特に、私たちのプロトタイプはバインディングブール式を導入します。これにより、パターンガード内、if 条件内（その後 then ブランチで使用）、while 条件内（ループ本体で使用）で変数をバインドできます。私たちのシステムはまた、スマートデストラクタ（スマートコンストラクタの双対）の定義を可能にする Haskell スタイルのビューも取り入れています。

このトークでは、評価規則、型付け規則、シンプルなコンパイルスキームを備えた ML に似た言語を紹介します。最後に実装のデモを行います：これらの構成要素を公開する拡張 ML 構文を解析し、私たちの（非常に直感的な！）規則に従ってプログラムを型チェックし、私たちの（非最適化）変換によって正しい OCaml AST に変換する OCaml PPX プロトタイプです。

OCaml の SIP サーバー、gRPC、HTTP/2 ライブラリのメンテナーを募集

Archive: https://discuss.ocaml.org/t/seeking-maintainers-for-our-ocaml-sip-server-grpc-and-http-2-libraries/18000/1

Wojtek Czekalski が発表しました

dialo で OCaml から移行するにあたり、私たちが構築したライブラリを適切にメンテナンスしてくれる方々に寄付したいと思っています。OCaml を使って非常に高性能なテレフォニーシステムを構築しましたが、小規模なチームとして維持するソフトウェアが多すぎました。

最初に OCaml で SIP サーバーの実装を構築しましたが、これは完璧なフィットでした。システムの残りの部分と言語非依存な方法で接続する必要がありました。gRPC を選択しましたが、そこから困難が始まりました。

小規模では h2 の上に構築した独自のバグだらけの gRPC 実装で十分でした。しかしスケールアップを始め、同時に OCaml 5.0 に移行しました。これらの要因の組み合わせにより、以下の問題が発生しました：

  1. アプリコード、SIP スタック、h2 内部でのメモリ管理に関連する 5.0 でのパフォーマンス低下。
  2. エラーとエッジケースをより適切に処理するためのより堅牢な gRPC 実装が必要でした。

結局：

  1. 独自の http2 実装を作成しました
  2. gRPC ライブラリを eio 専用に書き直し、まともなコード生成と堅牢性を備えました

その後、2025年後半に SIP サーバーに新しい機能を追加する必要があり、何かが心の中で折れました。SIP サーバーは 3.5 年かけてかなりのレガシーコードを積み重ねていたため、一部を書き直したかったのです。エフェクトを中心とした新しいアーキテクチャの構築に非常に興奮していましたが、ユーザースペースのエフェクトと並行処理ライブラリが実際にはうまく連携しないことを発見しました（コールバックがスコープを失うため）。それが最後の一撃でした。

Rust と Go でサービスを書き直す 1 週間のスプリントを 2 回実行することにしました。最終的に Rust を選択しました。まだ OCaml を恋しく思っていますが、独自の http2/grpc/sip スタックを実装・維持する必要がないのは清々しいです。

ということで — 私たちは徐々に OCaml をスタックから取り除いており、それが以下を寄付したい理由です：

  • ocaml-grpc - 新しいコードは dialo ブランチにあります。現在 eio のみ、本番環境でバトルテスト済み。いくつかのバグはありますが少なく、かなり安定しています。
  • haha (http2) - 全体的には良いですが改善の余地があります。ファイバーをキャンセルしすぎており、大幅に高速化できる非常に簡単な改善点です。
  • 私たちの SIP スタック — 現在はオープンソースではありませんが、適切な人やチームにはソースコードを喜んで共有します。

ppx_mixins：より洗練された mixin 構文

Archive: https://discuss.ocaml.org/t/ann-ppx-mixins-slightly-more-elegant-mixin-syntax/18001/1

Sacha Ayoun が発表しました

こんにちは。

ppx_mixins という小さな ppx を作りました。次のように書けます： 

type u [@@mixins Map.OrderedType + Printable]

これは以下に展開されます： 

type u
include Map.OrderedType with type t := u
include Printable with type t := u

大したことではありませんが、「mixin」パターンを多用するコードベースでの可読性が向上します。

型シグネチャの構築

近日公開予定のリリースでは、次のように書くこともできます： 

module type M = [%mixins Map.OrderedType + Printable]
(* 以下に展開されます *)
module type M = sig 
   type t 
   include Map.OrderedType with type t := t
   include Printable with type t := t
end

追加機能

他の型をオーバーライドすることもできます。例えば： 

type u [@@mixins Mappable(key = int; value := v)]
(* 以下に展開されます *)
type u
include Mappable with type t := u and type key = int and type value := v

制限事項

  • パラメトリック型のサポートなし（例：with type 'a u = 'a v）
  • タプル型と関数型のサポートなし（例：with type t = int -> bool）

プリプロセッサがペイロードを式として解析するため、これらは式として適切に解析されません。 型パラメータを持つ mixin のより深いサポートにはおそらく言語サポートも必要になるでしょう。

経験報告：Dune の依存関係グラフの改良

Archive: https://discuss.ocaml.org/t/experience-report-refining-dune-s-dependency-graph/18002/1

Robin Bate Boerop が発表しました

Dune の依存関係グラフの改良：モジュール単位のライブラリフィルタリング

Dune のライブラリ間依存関係追跡の改善に取り組んでおり、その経験を共有したいと思います。技術的な詳細と、この大規模なオープンソース OCaml プロジェクトへの初めての貢献者としての経験の両方についてです。

取り組んだ問題

libA が libB に依存する場合、Dune は libA のすべてのモジュールに libB のすべての .cmi ファイルへのグロブ依存関係を与えます。libB のいずれかの .cmi が変更されると、libA のすべてのモジュールが再コンパイルされます。たとえそのモジュールが libB を参照していなくても。

多くのライブラリを持つプロジェクトでは、これが不必要な再コンパイルの連鎖を生み出します。この問題を追跡するイシュー #4572 は 2021 年から開かれています。

私のアプローチ

Dune はすでにライブラリ内のモジュール依存関係を計算するために ocamldep を実行しています。重要な洞察：同じ出力がエントリモジュール名を介して各モジュールがどのライブラリを参照しているかを教えてくれます。これを使って、ビルド依存関係と -I~/-H~ コンパイラフラグの両方をモジュール単位でフィルタリングできます。

実装（PR #14116 と PR #14186）の仕組みは次の通りです：

  1. 各モジュールとそのライブラリ内の推移的依存関係について、ocamldep の出力（.ml と .mli の両方）を読み取る
  2. -open フラグを含む参照されるすべてのモジュール名を結合する
  3. それらの名前を Lib_index を介してライブラリにマッピングする
  4. Lib.closure を介してフィルタリングされたライブラリセットを推移的に閉じる
  5. 結果を非表示の deps と -I~/-H~ コンパイラフラグの両方に使用し、requires_compile のメンバーシップに基づいて直接（-I 経由で可視）と非表示（-H 経由）に分割する

deps とフラグの両方がフィルタリングされると、モジュールが宣言していないライブラリを参照する場合にクリーンビルドが失敗します。以前は過度に広い -I フラグがそのようなエラーをマスクする可能性がありました。

最初の失敗

最初の試み（PR #14021）は、テストカバレッジが不十分なまま単一の PR でフィルタリングを実装しようとしました。レビューで、私が予期していなかったエッジケース、特に透明モジュールエイリアスと仮想ライブラリに関するアプローチが脆弱であることが明らかになり、クローズされました。

課題

透明モジュールエイリアス。 OCaml のモジュールエイリアスメカニズムは、ocamldep がモジュールの推移的依存関係のすべてのライブラリを常に報告するわけではないことを意味します。libB が module M = LibC.Something を持ち、libA のモジュールが LibB.M を使用する場合、ocamldep は LibB を報告しますが LibC は報告しません。修正：コンパイルコンテキストで制限して Lib.closure を使用してフィルタリングされたライブラリセットを推移的に閉じます。

ルートモジュール。 (root_module) スタンザはコンパイルコンテキスト内のすべてのライブラリを暗黙的にエイリアスするモジュールを作成します。ocamldep がルートモジュールへの参照を報告する場合、どの基礎ライブラリが実際に必要かを判断できないため、完全な依存関係セットにフォールバックします。

仮想ライブラリ。 コンパイルコンテキストに仮想ライブラリの実装が存在する場合、パラメータライブラリが requires_compile に表示されないため、フィルタリングがそれらを見逃す可能性があります。別のフォールバックケースです。

Menhir で生成されたモジュール。 これらのモックモジュールは ocamldep の依存関係グラフにないため、それらのフィルタリングをスキップします。

nullビルドのオーバーヘッド。 フィルタリングはモジュール単位で .d ファイルを読み込み、ライブラリクロージャを計算します。新しい dune プロセス（メモキャッシュなし）では、これは変更がない場合も含めてすべてのビルドで新しい作業です。これは実際のトレードオフです：増分リビルドのパフォーマンス向上が nullビルドのオーバーヘッド増加の代償となります。

前提条件としてのテスト PR

実装 PR の前に、既存の動作を文書化し安全網を確立するためにテストのみの PR を 6 件提出しました：

  • #14017 — 現在のライブラリ間再コンパイル動作を文書化するベースラインテスト
  • #14031 — スタンザとライブラリ間のモジュール名シャドウイングを文書化するテスト
  • #14100 — コンパイルルールとサンドボックスビルドでのライブラリファイル deps を検証するテスト
  • #14101 — 透明エイリアスの増分ビルドの安全性を検証するテスト
  • #14129 — エイリアス再エクスポートライブラリを使用した増分ビルドを検証するテスト
  • #14178 — 透明エイリアスチェーンでの ocamldep の動作を文書化するテスト

これにより、実装 PR の差分が実際の変更に集中でき、レビュアーに既存の動作が保持されていることへの信頼を与えました。また、最初の試みがつまずいたエッジケースを理解するのにも役立ちました。

レビュープロセス

Dune のメンテナー（@rgrinberg と @art-w）は徹底的で建設的なレビューを提供してくれました。ハイライトとしては：

  • 私の手作りの推移的クロージャを既存のライブラリの Lib.closure に置き換えること — Dune の内部に精通していなければ見つけられなかったより洗練されたアプローチ
  • .ml と .mli の両方の ocamldep 出力を読む必要があることの特定（インターフェースが実装とは異なるライブラリを参照する可能性があるため）
  • モジュール単位の -I~/-H~ フラグフィルタリングの提案（クリーンビルドをより精密にし、キャッシングを改善する）
  • すべてのフォールバックケースと特殊ケースのモジュールタイプへの疑問、コードのシンプル化へ

PR はレビュー中に大幅なリファクタリングを経ました — 最終版は最初の提出よりも実質的にタイトです。

改善の余地

全体的に肯定的な経験でしたが、いくつかの点で摩擦がありました：

マージ前にベンチマークする方法がない。 nullビルドのオーバーヘッド問題はプロセスの後半で浮上しました。手動ベンチマークで、この変更が nullビルド時間に約 70% 追加されることを発見しました — これは重大な後退です。Dune のベンチマーク CI ワークフローは PR ではなく main へのプッシュ時のみ実行されます。貢献者がアクセスできるパフォーマンスツールがあれば、マージ前に後退を発見するのに役立つでしょう。

レビューのモメンタムとリベース。 テスト PR は素早くマージされましたが、実装 PR は数日間にわたって複数のレビューラウンドが必要でした。ラウンド間に main が進み、競合のリスクを持つリベースが必要になります。貢献者はブランチを新鮮に保つ負担を負います。これは PR が互いに依存している場合に複合します — #14116 のすべてのリベースには #14186 のリベースも必要でした。GitHub は PR スタックのファーストクラスサポートがないため、これは手動でエラーが起きやすいです。もちろん、すべての GitHub ホストのリポジトリがこの問題に悩んでいます。

不安定な CI。 多くの CI 実行で私のコードとは無関係のエラーがありました。一時的に到達できないか障害のある OCaml パッケージのアップストリームプロバイダーが原因であることが多かったです。これらの問題は自然に解決することが多かったですが、PR のライフタイムに日単位の遅延をもたらしました。問題は CI ジョブで繰り返し実行されるセットアップコードに起因します。

振り返り

Dune のコードベースはよく構造化されており、ビルドエンジン、ルール生成、スケジューラーの間の明確な分離があります。品質も高く、品質維持に費やす時間は価値があると感じさせます。

cram テストインフラはテストに適していると感じました。各テストシナリオは期待される出力を持つ自己完結型のシェルスクリプトであり、正確な再コンパイル動作を文書化して検証するのが容易です。コードへの信頼を与えます。

メンテナーは反応が良く、徹底さによって遅くなったレビュープロセスは協力的でプロフェッショナルです。メンテナーの皆さん、ありがとうございます！

ベースコンパイラは、opam 2.1 以降で以下のコマンドを使用して opam switch としてインストールできます： 

opam update
opam switch create 5.5.0~beta1

ベータのソースコードは以下のアドレスでも入手できます：

  • GitHub: https://github.com/ocaml/ocaml/archive/5.5.0-beta1.tar.gz
  • Inria の OCaml アーカイブ: https://caml.inria.fr/pub/distrib/ocaml-5.5/ocaml-5.5.0~beta1.tar.gz

細かいコンパイラ設定

コンパイラの設定を調整したい場合は、次のコマンドでオプションバリアントに切り替えられます： 

opam update
opam switch create <switch_name> ocaml-variants.5.5.0~beta1+options <option_list>

option_list はスペース区切りの ocaml-option-* パッケージのリストです。例えば、flambda と no-flat-float-array の switch の場合： 

opam switch create 5.5.0~beta1+flambda+nffa ocaml-variants.5.5.0~beta1+options ocaml-option-flambda ocaml-option-no-flat-float-array

利用可能なすべてのオプションは opam search ocaml-option で一覧できます。

上記の組み合わせにより、パラメータや戻り値の型にそれらの型を含む場合に以前は除外されていたメソッドをより多く生成できるようになり、GTK（および関連ライブラリ）の API サーフェスがより広く開かれます。

次のステップ：現在の焦点は、テストをより良く構造化し、gir_gen（GObject コードジェネレーター）を独立した dune プロジェクトに分割するための内部のクリーンアップと再編成です。これにより依存関係リストが大幅に削減され、以前の OCaml バージョンをターゲットにすることが可能になります。また、ocgtk を単に使用するパッケージには必要のない gir_gen に関連するリリースの頭痛も軽減されます。
