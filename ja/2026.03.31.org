* Merlinのsignature-help機能の改善
:PROPERTIES:
:CUSTOM_ID: 1
:END:
アーカイブ: https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1

** Charlène_Grosによるアナウンス


Taridesでは、Merlinの ~signature-help~ 機能にいくつかの改善を加えたことを大変嬉しくお知らせします!

念のため、~signature-help~ はMerlinの機能で、関数のシグネチャを表示し、アクティブなパラメータをハイライトします。エディタでは、通常ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。例えば、次の画像はVSCodeでの ~exec_prog~ 関数の ~signature-help~ を表示しています。

#+attr_html: :width 80%
https://us1.discourse-cdn.com/flex020/uploads/ocaml/original/2X/4/46e99c5c69bab991b8dd6a6a7494db5c2f32e07b.png

signature-helpコマンドの実装にはいくつかの欠点がありましたが、これらは修正されました。これらの修正により、この機能はずっと使いやすくなりました。これらの改善のいくつかは、最新のMerlinとocaml-lspのリリースにすでに含まれており、その他は次回のリリースに含まれる予定です。

結果:

- Merlinは関数パラメータに対してのみ ~signature-help~ 情報を表示し、関数名には表示しなくなりました。
- ~signature-help~ はアクティブなパラメータに対してのみ関数の情報を表示し、最後のパラメータの後に最初のパラメータをハイライトするようループバックしなくなりました。
- ~signature-help~ は、関数呼び出しの後に ~in~ がまだ書かれていない ~let .. in~ バインディング内で使用されても、トリガーされるようになりました。
- オプショナルパラメータが検出され、アクティブなパラメータがそのいずれかを書き始めるとハイライトされるようになりました。

* ortac-0.8 仕様駆動テストとDomains
:PROPERTIES:
:CUSTOM_ID: 4
:END:
アーカイブ: https://discuss.ocaml.org/t/ann-ortac-0-8-specification-driven-testing-with-domains/17927/1

** Nicolas Osborneによるアナウンス


こんにちは! Taridesでは、仕様駆動テストのためのortac-0.8.0のリリースを発表できることを大変嬉しく思います!

~ortac~ は[[https://github.com/ocaml-gospel][Gospel]]エコシステムのツールです。中核となるアイデアは、Gospel仕様言語のサブセットをOCamlコードに変換し、これらの変換を使用してランタイムチェックを生成することです。

~ortac~ はプラグインアーキテクチャを持っています。opam経由でインストールできます:

#+begin_example
$ opam install ortac-wrapper    # ランタイムアサーションチェックのためのwrapperプラグインを、必要な他のすべての部分(ランタイムとcli)とともにインストール
$ opam install ortac-qcheck-stm # QCheck-STMテストを生成するqcheck-stmプラグインを、必要な他のすべての部分(ランタイムとcli)とともにインストール
$ opam install ortac-dune       # 前述の2つのプラグインを使用するためのduneボイラープレートを生成するduneプラグインをインストール
#+end_example

今回のリリースでは、Ortac/QCheck-STMをQCheck-STMテストフレームワークのより多くの機能を活用できるようにすることに焦点を当てました:すなわち、並列コンテキストでのテストとコマンド生成の柔軟性です。

これがどのように動作するかを説明する[[https://ocaml-gospel.github.io/ortac/ortac-qcheck-stm/index.html][ドキュメント]]をご覧ください。

並列コンテキストでのテストに関して、バージョン0.2でのバグレポート機能の導入と、バージョン0.4でのSUT返却関数のカバレッジにより、並列コンテキストでQCheck-STMテストを簡単に生成することが困難になりました(手書きのQCheck-STMテストでは非常に簡単です)。これが今回修正されました!

バグレポートの情報は、実際のテストを行う関数(実際の実行結果とモデルからの結果を比較する)で部分的に収集されます。これは、並列テスト用のバグレポート機能を実装するには、この関数を書き直す必要があることを意味します。これはQCheck-STM+Domainsの中核であり、テストを実行する際に信頼しているものです(信頼されるテストベースと言えます)。同じ動作を維持していること、生成されたテストが対応する手書きのQCheck-STMテストと同じセマンティクスを持つことを確認するために、情報収集を導入した[[https://github.com/ocaml-gospel/ortac/commit/ef8d598ccbd6c9987648e201ebffb6abb5a3d610][コミット]]の差分を最小化し、ロジックの保存が明確になるようにしました(結局のところ、人間によるレビューが私たちが信頼するものです)。

バージョン0.4以降、SUT返却関数がテストに含まれるようになりました。新しく作成されたSUTは、次の呼び出しの引数として選択されるSUTのストアに追加されます。ここでの問題は、並列コンテキストでそれらをどう扱うかです:並列ブランチで作成されたSUTをグローバルストアに追加することはできません。ドメイン間で共有されることを想定していないからです。私たちはシンプルな設計を選択し、並列コンテキストに入ると新しく作成されたSUTの保存を停止します。心配しないでください。これらのSUT返却関数は、テストのシーケンシャル部分(シーケンシャルモードおよび/または並列モードのシーケンシャルプレフィックス)で完全にテストされます。

QCheck-STMテストフレームワークの強力な点の1つは、コマンドジェネレーターの柔軟性です。この柔軟性は、~QCheck.Gen~ API自体と、QCheck-STMコマンドジェネレーターが現在のモデルの状態でパラメータ化されているという事実から来ています(キー-バリューストアの ~lookup~ コマンドの生成を考えてみてください。実際に値に関連付けられているキーを検索する呼び出しを生成できる可能性が欲しいでしょう)。自動生成されたコマンドジェネレーターは少し素朴である必要があります。この素朴さを軽減するために、ユーザーが以下を提供できるようにしています:

- 特定のコマンド生成に適用される重み。例えば、キューを所有していないはずのドメインで[[https://ocaml-multicore.github.io/saturn/saturn/Saturn/Work_stealing_deque/index.html#val-push][work stealing queue]]の ~push~ コマンドの生成を無効にするのに便利です。
- ユーザーが、コマンドジェネレーターがモデルの現在の状態でパラメータ化されていることを活用したい場合の完全なコマンドジェネレーター実装。

ハッピーテスティング!
      



* OCaml 5.5.0の第3アルファリリース
:PROPERTIES:
:CUSTOM_ID: 5
:END:
アーカイブ: https://discuss.ocaml.org/t/third-alpha-release-for-ocaml-5-5-0/17928/1

** octachronによるアナウンス


OCaml 5.5.0の安定化作業が進んでおり、OCaml 5.5.0の第3アルファリリースを発表できることを嬉しく思います。

第2アルファは、再配置可能なコンパイラとブートストラップ間の予期しない相互作用により未リリースでした。この問題は今回の第3アルファで修正されました。

この重要な変更以外に、第1アルファと比較して、この新しいアルファリリースには以下が含まれます

- 2つのコード生成修正
- 3つの型システム修正
-  1つの標準ライブラリ修正

(完全なリストについては、以下の変更履歴を参照してください)。

全体として、5.5.0はかなり安定してきており、ほとんどの開発ツールのアルファバージョンが利用可能です。そのため、4月初めにベータリリースに切り替える予定です。

より具体的には、エコシステムの安定化に関する進行中の進捗状況は、[[https://github.com/ocaml/opam-repository/issues/29463][opamの5.5.0対応メタイシュー]]で追跡されています。

このアルファリリース用のコンパニオンアルファopamリポジトリも利用可能です
#+begin_example
$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git
#+end_example
これには、OCaml 5.5.0用に更新中のopamパッケージの進行中のアルファリリースが含まれています。

最終リリースは、まだ5月から6月の間を予定しています。

バグを見つけた場合は、[[https://github.com/ocaml/ocaml/issues][issue tracker]]で報告してください。

新機能とバグ修正の完全なリストに興味がある場合は、
[[https://github.com/ocaml/ocaml/blob/5.5/Changes][OCaml 5.5.0の更新された変更履歴]]が利用可能です。

ハッピーハッキング、
Florian Angeletti for the OCaml team

* CS6868 Concurrent Programming -- コース資料
:PROPERTIES:
:CUSTOM_ID: 7
:END:
アーカイブ: https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1

** KC Sivaramakrishnanによるアナウンス


現在、IIT Madrasで「CS6868: Concurrent Programming」というコースを教えています。このコースの目標は、OCaml 5を使用して並行および並列プログラミングを教え、その後、安全で高速な並列性のために[[https://oxcaml.org/][OxCaml]]を紹介することです。OCaml 5の部分をカバーし、次はOxCamlを開始する予定です。コミュニティとコース資料を共有するのが良いと思いました。

https://github.com/kayceesrk/cs6868_s26

CS6868はOCamlの知識を前提としており、これは[[https://github.com/kayceesrk/cs3100_m20][CS3100: Paradigms of Programming]]でカバーしています。このコースには、すべての講義、ノートブック、課題を含む[[https://www.youtube.com/watch?v=9R8Oim7YU20&list=PLt0HgEXFOHdkE-NTs87s7QjwYwqeihb-D][YouTubeプレイリスト]]が含まれています。

CS6868は、並列性の基礎、線形化可能性による正当性の推論、スピンロック、ミューテックス、条件変数を通じた並列プログラミング、メモリ一貫性モデル、パフォーマンスとノンブロッキングデータ構造の基本(ロックフリーのリンクリスト、キュー、スタック)を構築します。ここでは、コースは[[https://shop.elsevier.com/books/the-art-of-multiprocessor-programming/herlihy/978-0-12-415950-1][The Art of Multiprocessor Programming]]という本に密接に従っています。すべてのコードはOCaml 5で再実装されています。本の内容に加えて、コースはOCamlの緩和メモリモデルもカバーしています。また、[[https://github.com/ocaml-multicore/multicoretests/][qcheck-linとqcheck-stm]]を使用した並行プロパティベーステスト、[[https://ocaml.org/manual/5.3/tsan.html][TSANを使用したデータ競合検出]]など、OCaml固有のツールもかなりカバーしています。

これに続いて、コースは[[https://xavierleroy.org/control-structures/][Control structures in programming languages: from goto to algebraic effects]]の特定の部分に従って、effect handlerの基本をカバーします。その後、軽量スレッド、バッファ付きチャネル、選択的通信([[https://en.wikipedia.org/wiki/Concurrent_ML][Concurrent ML]]の流儀)、ネストされた並列プログラミング([[https://github.com/ocaml-multicore/domainslib][domainslib]]のように)、基本的な非同期IO([[https://github.com/ocaml-multicore/eio][Eio]]のように)を備えた、Goのような、マルチコア対応の本格的な並行プログラミングライブラリを構築します。

[[https://github.com/kayceesrk/cs6868_s26/blob/main/project_ideas.md][コースプロジェクト]]の増加するコレクションがあり、コース資料を学習すれば実行可能なはずです。コースプロジェクトのアイデアがあればお知らせください。

このコースは、過去に行った[[https://github.com/ocaml-multicore/ocaml-effects-tutorial][多くの]] [[https://github.com/ocaml-multicore/ocaml5-parallelism-tutorial][チュートリアル]]の[[https://github.com/ocaml-multicore/parallel-programming-in-multicore-ocaml][内容]]を[[https://github.com/ocaml-multicore/effects-examples][まとめる]]ことを試みています。フィードバックがあればお知らせください。小さな問題を修正するPRは喜んで受け入れますが、これは進行中のコースなので、大きなPRは受け入れられない可能性が高いです。私がそれを作るのを楽しんだように、皆さんもこれに取り組むことを楽しんでいただけることを願っています。
