* Merlin の signature-help 機能の改善
:PROPERTIES:
:CUSTOM_ID: 1
:END:
Archive: https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1

** Charlène_Gros によるアナウンス


Tarides の私たちは、Merlin の ~signature-help~ 機能に加えられた改善点を発表できることを大変嬉しく思います！

念のため説明しますと、~signature-help~ は Merlin の機能で、関数シグネチャを表示し、アクティブなパラメータをハイライトします。エディタ上では、通常ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。以下の画像は、VSCode で関数 ~exec_prog~ の ~signature-help~ を表示した例です。

#+attr_html: :width 80%
https://us1.discourse-cdn.com/flex020/uploads/ocaml/original/2X/4/46e99c5c69bab991b8dd6a6a7494db5c2f32e07b.png

signature-help コマンドの実装にはいくつかの問題点があり、それらが修正されました。これらの修正により、機能がより快適に使えるようになりました。改善点の一部はすでに最新の Merlin および ocaml-lsp リリースに含まれており、その他は次のリリースに含まれる予定です。

修正内容：

- Merlin は関数のパラメータにのみ ~signature-help~ 情報を表示するようになり、関数名には表示しなくなりました。
- ~signature-help~ はアクティブなパラメータの情報のみを表示するようになり、最後のパラメータの後に最初のパラメータに戻るループが発生しなくなりました。
- 関数呼び出しの後に ~in~ がまだ書かれていない ~let .. in~ バインディングの内部でも ~signature-help~ がトリガーされるようになりました。
- アクティブなパラメータとしてオプションパラメータの記述が始まったとき、そのオプションパラメータが検出されハイライトされるようになりました。

* ortac-0.8 - Domains を用いた仕様駆動テスト
:PROPERTIES:
:CUSTOM_ID: 4
:END:
Archive: https://discuss.ocaml.org/t/ann-ortac-0-8-specification-driven-testing-with-domains/17927/1

** Nicolas Osborne によるアナウンス


こんにちは！Tarides の私たちは、仕様駆動テストのための ortac-0.8.0 のリリースを発表できることを大変嬉しく思います！

~ortac~ は [[https://github.com/ocaml-gospel][Gospel]] エコシステムのツールです。核心となるアイデアは、Gospel 仕様言語のサブセットを OCaml コードに変換し、その変換を使ってランタイムチェックを生成することです。

~ortac~ はプラグインアーキテクチャを採用しています。opam 経由でインストールできます：

#+begin_example
$ opam install ortac-wrapper    # install the wrapper plugin for runtime assertion checking, with all the other necessary parts (runtime and cli)
$ opam install ortac-qcheck-stm # install the qcheck-stm plugin that generates QCheck-STM tests, with all the other necessary parts (runtime and cli)
$ opam install ortac-dune       # install the dune plugin to generate dune boilerplate to use the two previous plugins
#+end_example

このリリースは、Ortac/QCheck-STM が QCheck-STM テストフレームワークのより多くの機能を活用できるようにすることに焦点を当てています。具体的には、並列コンテキストでのテストとコマンド生成の柔軟性です。

動作の仕組みについては[[https://ocaml-gospel.github.io/ortac/ortac-qcheck-stm/index.html][ドキュメント]]をご覧ください。

並列コンテキストでのテストについて：バージョン 0.2 でのバグレポート機能の導入とバージョン 0.4 での SUT を返す関数のカバレッジにより、並列コンテキストでの QCheck-STM テストの生成が容易ではなくなっていました（手書きの QCheck-STM テストでは比較的簡単です）。この問題が修正されました。

バグレポートの情報は、実際のテスト（実際の実行結果とモデルの結果を比較する）を行う関数内で部分的に収集されます。これは、並列テスト用のバグレポート機能を実装するには、QCheck-STM+Domains の核心部分でありテスト実行時に信頼している関数を書き直す必要があることを意味します。同じ動作を保持し、生成されたテストが対応する手書きの QCheck-STM テストと同じセマンティクスを持つことを確認するために、情報収集を導入する[[https://github.com/ocaml-gospel/ortac/commit/ef8d598ccbd6c9987648e201ebffb6abb5a3d610][コミット]]の差分を最小化し、ロジックの保存が明確になるようにしました（最終的には、人間によるレビューが信頼の拠り所です）。

バージョン 0.4 以降、SUT を返す関数がテストに含まれるようになっています。新しく作成された SUT は、次の呼び出しの引数として選択される SUT のストアに追加されます。ここでの問題は、並列コンテキストでそれらをどう扱うかです：並列ブランチで作成された SUT はドメイン間で共有されるべきではないため、グローバルなストアに追加することはできません。シンプルな設計として、並列コンテキストに入ったら新しく作成された SUT のストア追加を停止することにしました。ご安心ください、これらの SUT を返す関数は、テストの逐次部分（逐次モードおよび/または並列モードの逐次プレフィックス）で引き続き完全にテストされます。

QCheck-STM テストフレームワークの強みの一つは、コマンドジェネレータの柔軟性です。この柔軟性は ~QCheck.Gen~ API 自体と、QCheck-STM コマンドジェネレータが現在のモデルの状態に対してパラメータ化されているという事実から来ています（キーバリューストアの ~lookup~ コマンドの生成を考えてみてください。実際に値と関連付けられているキーを検索するコールを生成できる確率を持たせたいでしょう）。自動生成されたコマンドジェネレータはやや単純になりがちです。この単純さを軽減するために、ユーザーが以下を提供できるようにしました：

- 特定のコマンド生成に適用するウェイト。例えば、[[https://ocaml-multicore.github.io/saturn/saturn/Saturn/Work_stealing_deque/index.html#val-push][work stealing queue]] のキューを所有すべきでないドメインでの ~push~ コマンドの生成を無効化するのに有用です。
- コマンドジェネレータが現在のモデルの状態に対してパラメータ化されることを活用したい場合の、完全なコマンドジェネレータの実装。

テストを楽しんでください！
      



* OCaml 5.5.0 の第3アルファリリース

** octachron によるアナウンス


OCaml 5.5.0 の安定化作業の進展に伴い、OCaml 5.5.0 の第3アルファリリースを発表できることを嬉しく思います。

第2アルファは、再配置可能コンパイラとブートストラップの間に予期しない相互作用が発見されたためリリースされませんでした。この問題は第3アルファで修正されました。

この重要な変更に加え、第1アルファと比べてこの新しいアルファリリースには以下が含まれています：

- コード生成の修正 2件
- 型システムの修正 3件
- 標準ライブラリの修正 1件

（完全なリストは以下の変更履歴をご覧ください）

全体的に、5.5.0 の安定化は順調に進んでいるようで、ほとんどの開発ツールのアルファ版が利用可能です。そのため、4月上旬にはベータリリースに切り替える予定です。

具体的には、エコシステムの安定化の進行状況は [[https://github.com/ocaml/opam-repository/issues/29463][opam 上の 5.5.0 準備メタ Issue]] で追跡されています。

このアルファリリース用のコンパニオン alpha opam リポジトリも利用可能です：
#+begin_example
$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git
#+end_example
このリポジトリには OCaml 5.5.0 向けに更新中の opam パッケージの進行中のアルファリリースが含まれています。

最終リリースは引き続き5月から6月の間を予定しています。

バグを発見した場合は、[[https://github.com/ocaml/ocaml/issues][Issue トラッカー]]に報告してください。

新機能とバグ修正の完全なリストに興味がある場合は、更新された [[https://github.com/ocaml/ocaml/blob/5.5/Changes][OCaml 5.5.0 の変更履歴]] をご覧ください。

Happy hacking,
OCaml チームを代表して Florian Angeletti

* CS6868 並行プログラミング -- コース教材
:PROPERTIES:
:CUSTOM_ID: 7
:END:
Archive: https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1

** KC Sivaramakrishnan によるアナウンス


現在、IIT Madras で「CS6868: 並行プログラミング」というコースを教えています。このコースの目標は、OCaml 5 を使って並行・並列プログラミングを教え、その後、安全で高速な並列処理のために [[https://oxcaml.org/][OxCaml]] を紹介することです。OCaml 5 の部分はすでに終わり、次は OxCaml を始める予定です。コミュニティとコース教材を共有することが良いと思いました。

https://github.com/kayceesrk/cs6868_s26

CS6868 は OCaml の知識を前提としており、[[https://github.com/kayceesrk/cs3100_m20][CS3100: プログラミングパラダイム]]でカバーしています。このコースには、すべての講義、ノートブック、課題を含む[[https://www.youtube.com/watch?v=9R8Oim7YU20&list=PLt0HgEXFOHdkE-NTs87s7QjwYwqeihb-D][YouTube プレイリスト]]があります。

CS6868 では、並列性の基礎、線形化可能性による正確性の推論、スピンロック・ミューテックス・条件変数を通じた並列プログラミング、メモリ整合性モデル、パフォーマンスの基礎とノンブロッキングデータ構造（ロックフリー連結リスト、キュー、スタック）を構築します。ここでは、[[https://shop.elsevier.com/books/the-art-of-multiprocessor-programming/herlihy/978-0-12-415950-1][The Art of Multiprocessor Programming]] という書籍に密接に従っています。すべてのコードは OCaml 5 で書き直されています。書籍の内容に加え、コースでは OCaml の緩和メモリモデルもカバーしています。また、[[https://github.com/ocaml-multicore/multicoretests/][qcheck-lin と qcheck-stm]] を使った並行プロパティベーステストや、[[https://ocaml.org/manual/5.3/tsan.html][TSAN を使ったデータ競合検出]]など、OCaml 固有のツールも相当量カバーしています。

その後、[[https://xavierleroy.org/control-structures/][Control structures in programming languages: from goto to algebraic effects]] の特定の部分に従って、エフェクトハンドラーの基礎をカバーします。そして、軽量スレッド、バッファードチャネル、選択的通信（[[https://en.wikipedia.org/wiki/Concurrent_ML][Concurrent ML]] 的な）、ネスト並列プログラミング（[[https://github.com/ocaml-multicore/domainslib][domainslib]] のような）、基本的な非同期IO（[[https://github.com/ocaml-multicore/eio][Eio]] のような）を備えた、本格的な Go 風マルチコア対応並行プログラミングライブラリを構築します。

コース教材を通じて取り組めば達成可能な[[https://github.com/kayceesrk/cs6868_s26/blob/main/project_ideas.md][コースプロジェクト]]も増えています。コースプロジェクトのアイデアがあればお知らせください。

このコースは、過去に行った[[https://github.com/ocaml-multicore/ocaml-effects-tutorial][多くの]][[https://github.com/ocaml-multicore/ocaml5-parallelism-tutorial][チュートリアル]][[https://github.com/ocaml-multicore/parallel-programming-in-multicore-ocaml][の]][[https://github.com/ocaml-multicore/effects-examples][内容]]をまとめることを目指しています。フィードバックがあればお知らせください。小さな問題を修正する PR は受け付けますが、現在進行中のコースであるため、大きな PR はおそらく受け付けられません。皆さんが私が作るのと同じくらい楽しんで取り組んでいただけることを願っています。
