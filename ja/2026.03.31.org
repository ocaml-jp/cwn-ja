* MerlinのSignature-help機能の改善
:PROPERTIES:
:CUSTOM_ID: 1
:END:
Archive: https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1

** Charlène_Gros が発表しました


Taridesの私たちは、Merlinの~signature-help~機能に加えられたいくつかの改善をお知らせできることを大変嬉しく思います！

念のため説明しますと、~signature-help~はMerlinの機能で、関数のシグネチャを表示してアクティブなパラメータをハイライトするものです。エディタでは通常、ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。たとえば、以下の画像はVSCodeにおける関数~exec_prog~の~signature-help~を示しています。

#+attr_html: :width 80%
https://us1.discourse-cdn.com/flex020/uploads/ocaml/original/2X/4/46e99c5c69bab991b8dd6a6a7494db5c2f32e07b.png

signature-helpコマンドの実装にはいくつかの問題点があり、それらが修正されました。これらの修正により、機能がより使いやすくなります。一部の改善はすでに最新のMerlinおよびocaml-lspのリリースに含まれており、残りは次のリリースに含まれる予定です。

改善内容：

- Merlinは関数のパラメータ上でのみ~signature-help~情報を表示するようになり、関数名では表示されなくなりました。
- ~signature-help~はアクティブなパラメータに対してのみ関数情報を表示するようになり、最後のパラメータの後に最初のパラメータに戻ってハイライトするループがなくなりました。
- ~let .. in~バインディングの内部で使用された場合、関数呼び出しの後に~in~がまだ書かれていなくても~signature-help~がトリガーされるようになりました。
- アクティブなパラメータがオプショナルパラメータの書き込みを開始すると、オプショナルパラメータが検出されてハイライトされるようになりました。

* ortac-0.8 Domainsを使った仕様駆動テスト
:PROPERTIES:
:CUSTOM_ID: 4
:END:
Archive: https://discuss.ocaml.org/t/ann-ortac-0-8-specification-driven-testing-with-domains/17927/1

** Nicolas Osborne が発表しました


こんにちは！Taridesの私たちは、仕様駆動テストのためのortac-0.8.0のリリースを発表できることをとても興奮しています！

~ortac~は[[https://github.com/ocaml-gospel][Gospel]]エコシステムのツールです。核心的なアイデアは、Gospelの仕様言語のサブセットをOCamlコードに変換し、これらの変換を使ってランタイムチェックを生成することです。

~ortac~はプラグインアーキテクチャを採用しています。opam経由でインストールできます：

#+begin_example
$ opam install ortac-wrapper    # install the wrapper plugin for runtime assertion checking, with all the other necessary parts (runtime and cli)
$ opam install ortac-qcheck-stm # install the qcheck-stm plugin that generates QCheck-STM tests, with all the other necessary parts (runtime and cli)
$ opam install ortac-dune       # install the dune plugin to generate dune boilerplate to use the two previous plugins
#+end_example

このリリースは、Ortac/QCheck-STMがQCheck-STMテストフレームワークのより多くの機能、すなわち並列コンテキストでのテストとコマンド生成の柔軟性を活用できるようにすることに焦点を当てています。

動作の仕組みについては[[https://ocaml-gospel.github.io/ortac/ortac-qcheck-stm/index.html][ドキュメント]]をご覧ください。

並列コンテキストでのテストについては、バージョン0.2でのバグレポート機能の導入とバージョン0.4でのSUT返却関数のカバレッジにより、並列コンテキストでのQCheck-STMテストの生成が（手書きのQCheck-STMテストでは非常に簡単であるにもかかわらず）容易ではなくなっていました。これが修正されました！

バグレポートの情報は、実際のテスト（実際の実行結果とモデルからの結果を比較する）を行う関数で部分的に収集されます。つまり、並列テストにバグレポート機能を実装するには、QCheck-STM+Domainsの核心であり、テストを実行する際に信頼している関数（いわばTrusted Testing Base）を書き直す必要があります。同じ動作が維持されること、生成されたテストが対応する手書きのQCheck-STMテストと同じセマンティクスを持つことを確認するために、ロジックの保存が明らかになるよう情報収集を導入した[[https://github.com/ocaml-gospel/ortac/commit/ef8d598ccbd6c9987648e201ebffb6abb5a3d610][コミット]]のdiffを最小化しました（最終的には、人間によるレビューが信頼の根拠です）。

バージョン0.4以降、SUT返却関数はテストに含まれています。新しく作成されたSUTは、次の呼び出しの引数として選択されるSUTのストアに追加されます。ここでの問題は、並列コンテキストでこれらをどう扱うかです：並列ブランチで作成されたSUTをグローバルストアに追加することはできません。ドメイン間で共有されることを想定していないからです。シンプルな設計を選択し、並列コンテキストに入ったら新しく作成されたSUTのストアへの保存を停止します。これらのSUT返却関数は、テストのシーケンシャル部分（シーケンシャルモードおよび/または並列モードのシーケンシャルプレフィックス）で引き続き完全にテストされますのでご安心ください。

QCheck-STMテストフレームワークの強みの一つは、コマンドジェネレーターの柔軟性です。この柔軟性は~QCheck.Gen~APIそのものと、QCheck-STMコマンドジェネレーターが現在のモデルの状態にパラメータ化されているという事実から来ています（キーバリューストアの~lookup~コマンドの生成を考えてみてください。実際に値が関連付けられているキーを検索する呼び出しを生成するチャンスを得たいはずです）。自動生成されたコマンドジェネレーターはやや素朴になりがちです。この素朴さを緩和するために、ユーザーが以下を提供できるようにしています：

- 特定のコマンド生成に適用するウェイト。例えば、[[https://ocaml-multicore.github.io/saturn/saturn/Saturn/Work_stealing_deque/index.html#val-push][work stealing queue]]でキューを所有していないドメインでの~push~コマンドの生成を無効化するのに有用です。
- コマンドジェネレーターがモデルの現在の状態にパラメータ化されていることを活用したい場合の完全なコマンドジェネレーター実装。

ハッピーテスティング！
      



* OCaml 5.5.0 の第3アルファリリース
:PROPERTIES:
:CUSTOM_ID: 5
:END:
Archive: https://discuss.ocaml.org/t/third-alpha-release-for-ocaml-5-5-0/17928/1

** octachron が発表しました


OCaml 5.5.0の安定化作業の進展に伴い、OCaml 5.5.0の第3アルファリリースを発表できることを嬉しく思います。

欠番となった第2アルファは、再配置可能なコンパイラとブートストラップの間の予期せぬ相互作用により未リリースとなりました。この問題は第3アルファで修正されています。

この重要な変更に加えて、第1アルファとの比較でこの新しいアルファリリースには以下が含まれています：

- コード生成のバグ修正が2件
- 型システムのバグ修正が3件
- 標準ライブラリのバグ修正が1件

（完全なリストについては以下のChangelogをご覧ください。）

全体的に、5.5.0の安定化は順調に進んでいるようで、ほとんどの開発ツールのアルファバージョンが利用可能です。そのため、4月初旬にベータリリースに切り替える予定です。

より詳しくは、エコシステムの安定化の進捗状況は[[https://github.com/ocaml/opam-repository/issues/29463][opamの5.5.0 readiness meta-issue]]で追跡されています。

このアルファリリース用のコンパニオンアルファopamリポジトリも利用可能です：
#+begin_example
$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git
#+end_example
これにはOCaml 5.5.0向けに更新中のopamパッケージの進行中のアルファリリースが含まれています。

最終リリースは引き続き5月から6月の間を予定しています。

バグを見つけた場合は、[[https://github.com/ocaml/ocaml/issues][issue tracker]]にご報告ください。

新機能とバグ修正の完全なリストに興味がある方は、更新された[[https://github.com/ocaml/ocaml/blob/5.5/Changes][OCaml 5.5.0のchangelog]]をご覧ください。

Happy hacking,
Florian Angeletti（OCamlチームを代表して）

* CS6868 並行プログラミング — コース資料
:PROPERTIES:
:CUSTOM_ID: 7
:END:
Archive: https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1

** KC Sivaramakrishnan が発表しました


現在、IIT Madrasで「CS6868: Concurrent Programming（並行プログラミング）」というコースを教えています。このコースの目標は、OCaml 5を使って並行・並列プログラミングを教え、その後、安全で高速な並列処理のために[[https://oxcaml.org/][OxCaml]]を紹介することです。OCaml 5の部分は済みましたので、次はOxCamlを始める予定です。コース資料をコミュニティと共有するのが良いと思いました。

https://github.com/kayceesrk/cs6868_s26

CS6868はOCamlの知識を前提としており、それは[[https://github.com/kayceesrk/cs3100_m20][CS3100: Paradigms of Programming]]でカバーしています。このコースには全講義のノートブックと課題を含む[[https://www.youtube.com/watch?v=9R8Oim7YU20&list=PLt0HgEXFOHdkE-NTs87s7QjwYwqeihb-D][YouTubeプレイリスト]]があります。

CS6868は並列性の基礎、線形化可能性による正確性の推論、スピンロック・ミューテックス・条件変数・メモリ一貫性モデルによる並列プログラミング、パフォーマンスの基礎とノンブロッキングデータ構造（ロックフリー連結リスト、キュー、スタック）を扱います。ここでは、コースは[[https://shop.elsevier.com/books/the-art-of-multiprocessor-programming/herlihy/978-0-12-415950-1][The Art of Multiprocessor Programming]]という本に密接に従っています。すべてのコードはOCaml 5で書き直されています。本のコンテンツに加え、コースではOCamlの緩和メモリモデルもカバーしています。また、[[https://github.com/ocaml-multicore/multicoretests/][qcheck-linとqcheck-stm]]を使った並行プロパティベーステストや、[[https://ocaml.org/manual/5.3/tsan.html][TSANを使ったデータレース検出]]など、OCaml固有のツールも相当量カバーしています。

その後、[[https://xavierleroy.org/control-structures/][Control structures in programming languages: from goto to algebraic effects]]の特定の部分に従って、エフェクトハンドラーの基礎をカバーします。続いて、軽量スレッド・バッファードチャンネル・選択的通信（[[https://en.wikipedia.org/wiki/Concurrent_ML][Concurrent ML]]の流れで）・ネスト並列プログラミング（[[https://github.com/ocaml-multicore/domainslib][domainslib]]のように）・基本的な非同期IO（[[https://github.com/ocaml-multicore/eio][Eio]]のように）を備えた本格的なGoのようなマルチコア対応並行プログラミングライブラリを構築します。

[[https://github.com/kayceesrk/cs6868_s26/blob/main/project_ideas.md][コースプロジェクト]]のコレクションも増えており、コース資料を通読すれば実施可能なものです。コースプロジェクトのアイデアがあればお知らせください。

このコースは、過去に行った[[https://github.com/ocaml-multicore/ocaml-effects-tutorial][多くの]][[https://github.com/ocaml-multicore/ocaml5-parallelism-tutorial][チュートリアル]][[https://github.com/ocaml-multicore/parallel-programming-in-multicore-ocaml][から]][[https://github.com/ocaml-multicore/effects-examples][のコンテンツ]]をまとめようとしています。フィードバックがあればお知らせください。小さな問題を修正するPRは受け付けますが、進行中のコースであるため、大きなPRは受け付けられない可能性が高いです。作成するのと同じくらい楽しんで取り組んでいただければ幸いです。
      



* libinput - Linuxの入力デバイス向けOCamlバインディング
:PROPERTIES:
:CUSTOM_ID: 8
:END:
Archive: https://discuss.ocaml.org/t/ann-libinput-ocaml-bindings-for-linux-input-devices/17933/1

** Thomas Leonard が発表しました


[[https://github.com/talex5/libinput-ocaml][libinput-ocaml]]（libinput CライブラリのOCamlバインディング）の最初のリリースを発表できることを嬉しく思います。

libinputは、Waylandコンポジターなどのアプリケーションがマウス、キーボード、タッチパッドなどにアクセスするために使用されます。
