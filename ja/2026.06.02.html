doctor 0.3.0: 読み取り専用の OCaml 環境診断ツール

Archive: https://discuss.ocaml.org/t/ann-doctor-0-3-0-read-only-ocaml-environment-diagnostics/18186/1

Funwithcthulhu が発表しました

こんにちは、

doctor 0.3.0 をリリースしました。

doctor は、ローカルの OCaml 開発環境をチェックするための小さな読み取り専用 CLI ツールです。opam、スイッチ、PATH、dune、ocaml-lsp-server、ocamlformat、VS Code OCaml Platform 拡張機能に関するよくあるセットアップの問題を報告します。

インストール： 

opam update
opam install doctor

基本的な使い方： 

doctor check
doctor check --json

0.3.0 リリースでは、opam スイッチ環境の診断に重点を置いています。特に、ツールが見つからない場合と、opam にアクティブなスイッチはあるが現在のシェルがそのスイッチ環境を読み込んでいない場合を区別しようとしています。

たとえば、ocaml がアクティブなスイッチの外で解決される場合や、dune、ocamllsp、ocamlformat などのインストール済みスイッチツールが PATH に含まれていない場合に報告できます。

このコマンドは opam init の実行、スイッチの作成、パッケージのインストール、シェル・エディタファイルの編集は一切行いません。見つかった内容を報告し、手動で実行すべきコマンドを提案するだけです。

JSON 出力は、安定した診断名とエラーコードのマッピングを必要とするエディタ統合や小さなスクリプト向けです。

リポジトリ：

https://github.com/funwithcthulhu/doctor

opam パッケージ：

https://opam.ocaml.org/packages/doctor/

わかりにくい診断内容や、よくあるセットアップケースで不足しているものについてのフィードバックは大歓迎です。

ありがとうございました！

Solo5: ユニカーネル向けサンドボックス実行環境

Archive: https://discuss.ocaml.org/t/ann-solo5-a-sandboxed-execution-environment-for-unikernels/18191/1

Calascibetta Romain が発表しました

Solo5.0.11.0 のリリースをお知らせできることを大変うれしく思います。このリリースは、OCaml におけるユニカーネルのパフォーマンス改善の第一歩として重要な意味を持ちます。

プロジェクト自体は C で書かれていますが、ユニカーネル開発の要となっています。OCaml のサポートは ocaml-solo5（OCaml コンパイラのバージョンに追従）を通じて提供されています。以下は、OCaml を使った Solo5（hvt）ユニカーネルの簡単な例です： 

$ opam install ocaml-solo5
$ eval $(opam env)
$ cat >startup.c<<EOF
#include "solo5.h"
#include <caml/callback.h>

static char *argv[] = { "unikernel", NULL };

void _nolibc_init(uintptr_t, size_t);

int solo5_app_main(const struct solo5_start_info *si) {
  _nolibc_init(si->heap_start, si->heap_size);
  caml_startup(argv);
  return (0);
}
EOF
$ cat >manifest.json<<EOF
{
  "type": "solo5.manifest",
  "version": 1,
  "devices": []
}
EOF
$ cat >main.ml<<EOF
let () = print_endline "Hello World!"
EOF
$ solo5-elftool gen-manifest manifest.json manifest.c
$ ocamlfind -toolchain solo5 opt \
  manifest.c startup.c main.ml \
  -cclib "-z solo5-abi=hvt" -o main.hvt
$ solo5-hvt -- main.hvt

OCaml でのユニカーネル開発についてさらに詳しく知りたい方は、スライド（最後の MirageOS リトリートでの発表）や、mnet のチュートリアル（こちら）をご覧ください。

Solo5 とは？

Solo5 は、サンドボックス内でプログラムを作成・実行するための C フレームワークとツールセットです。当初は外部世界との相互作用を以下の5つだけに制限しています：

  1. net デバイスへの Ethernet フレームの読み書き
  2. block デバイスへのページの読み書き
  3. 終了

サンドボックス環境で利用可能なすべての関数は、このヘッダーファイルに記述されています。

Solo5 の注目すべき点は、この API が VirtIO、Xen（および QubesOS）、そして独自のミニマリストなハイパーバイザー hvt（KVM、BHyve、VMM 上で動作）で動作することです。つまり、Solo5 アプリケーションは幅広いプラットフォームにデプロイできます。Ubuntu では、以下のコマンドで Solo5 をインストールできます： 

$ curl -fsSL https://apt.robur.coop/gpg.pub | \
  gpg --dearmor > /etc/apt/trusted.gpg.d/apt.robur.coop.gpg
$ echo "deb [signed-by=/etc/apt/trusted.gpg.d/apt.robur.coop.gpg] https://apt.robur.coop ubuntu-24.04 main" > /etc/apt/sources.list.d/robur.list
$ apt update
$ apt install solo5

hvt ユニカーネル

Robur では、hvt ユニカーネルのビルドを好んでいます。これらのユニカーネルは、Linux、FreeBSD、OpenBSD で利用可能な solo5-hvt 「テンダー」（約200KB）だけで実行できます（WSL はいわゆる ネスト 仮想化をサポートしているため、WSL 上でユニカーネルを実行することも可能です！）。

hvt の利点は、そのミニマリズム（Solo5 全般の特徴）にあります。今回のリリースでは、libseccomp を使用してユニカーネルの実行に必要な権限だけを残すことで、Linux 上のサンドボックスが完成しました（合計18のシステムコールに相当）。このサンドボックスは、FreeBSD では Capsicum、OpenBSD では pledge(2) によってすでに利用可能でした。

このリリースでは、eventfd(2) と tap インターフェースへ直接書き込む並列スレッドを使用することにより、Linux 上の tap インターフェースへの Ethernet フレーム書き込みのスループットも向上しています。これにより、コストのかかる VM 退出を減らすことができます。

そのために、iperf3（および mnet でこのプロトコルを実装したユニカーネル）を使ったベンチマークを実施し、スループットの改善を示しました。

solo5.0.10.1 solo5.0.11.0
~700 Mbits/sec ~2.15 Gbits/sec

ユニカーネルのデプロイ

Robur では、ユニカーネルのデプロイをできる限り簡単にすることを目指しています。そのため、様々なサービスをデプロイするためのソフトウェアとユニカーネルのスイートを開発しました：

  1. Albatross は（apt でもインストール可能な）プロジェクトで、hvt ユニカーネルを起動・監視できるデーモンを実行できます。このサービスは hvt ユニカーネル専用に設計されていますが（最近ではシステムもデプロイできるようになっています）、その点が強みです。
  2. dnsvizor は、IPアドレス解決とDNS解決の問題に対処する私たちのユニカーネルの一つです。このユニカーネルを起動することで、後続のユニカーネルがそれを基準にネットワークを設定できます。
  3. 最後に、mollymawk を積極的に開発しています。これは、新しいユニカーネルをデプロイするための Albatross 向けウェブインターフェースを提供するユニカーネルです（できる限り簡単に）。

また、aussi（まだ非常に実験的な段階）も開発しています。これは私たちのユニカーネル向けの OCI ランタイムです。中でも、Docker を使ってユニカーネルをビルド・デプロイできるようになります。私たちのユニカーネル annuaire を使った例はこちらでご覧いただけます。

ユニカーネルの開発

最後に、ユニカーネルの開発という側面があります。Robur では、新しいユーザーが自分自身で新しいユニカーネルを開発する手助けをするためのライブラリ（特にプロトコルやフォーマット）を開発しようとしています（オーナーシップの感覚が生まれるように）。

ぜひお気軽にご質問ください！

この観点から、私たちは標準的なユニカーネルの開発を可能にする多くのライブラリを長年にわたって開発してきました：

  • ユニカーネル（およびサービス）向けスケジューラとして Miou の開発を継続しています
  • ユニカーネル向けの新しい TCP/IP スタックである mnet と utcp を開発しています
  • ユニカーネル向け FAT32 ファイルシステム mfat も利用可能です
  • ウェブサイトをユニカーネルとして開発するための Web フレームワーク vifu も利用可能です
  • その他にも、長年メンテナンスしてきた多くのプロジェクトがあります！

まとめ

このリリースは、私たちの協同組合の取り組みについて包括的（かつわかりやすい）概要をお伝えする機会となりました。この取り組みの本質は、OCaml でユニカーネルを開発・デプロイするために必要なコンポーネントを作成しながら、コミュニケーションと生産手段の主権を取り戻すという考えを大切にすることです。

Happy hacking!

OCaml Workshop 2026 トーク提案募集

Archive: https://discuss.ocaml.org/t/call-for-talk-proposals-ocaml-workshop-2026/18192/1

Sudha Parimala が発表しました

みなさん、こんにちは！今年の OCaml Workshop はパリで開催されます。ワークショップへの発表提案を募集できることを嬉しく思います。

重要な情報

  • 開催地: パリ（FPW 2026 と共同開催）
  • 日程: 8月27日（変更の可能性あり！ 代替案：8月31日）
  • 提案締め切り: 7月1日（AoE）
  • 発表者への通知: 7月27日
  • 提案サイト: https://types-hotcrp.paris.inria.fr/ocaml26/

スコープ

OCaml Workshop は、OCaml の魔法を発見しつつある愛好家から、危険な呪文を熟知したウィザードまで、幅広い OCaml ユーザーを対象としています。共通点は OCaml への情熱と、より多くを学び、仲間の OCamler とつながり、言語を改善する方法を共に見つけていくことへの意欲です。

発表提案も同様に幅広く歓迎します：OCaml に関係するものであれば何でも歓迎です！

テーマは以下に限定されませんが、過去の例を挙げると：OCaml 編集ツール、検証済み OCaml 成果物、OCaml と他言語の相互運用性、OCaml の行動規範、コンパイラ最適化、OS 移植性、OCaml テストフレームワーク、OCaml の並行性パッケージなど、さまざまなテーマが発表されてきました。

過去のエディションの全カタログは以下のリンクからご覧いただけます：

  • 2025: https://ocaml.org/conferences/ocaml-workshop-2025
  • 2024: https://ocaml.org/conferences/ocaml-workshop-2024
  • 2023: https://ocaml.org/conferences/ocaml-workshop-2023
  • 2022: https://ocaml.org/conferences/ocaml-workshop-2022
  • 2021: https://ocaml.org/conferences/ocaml-workshop-2021

形式

20分の標準トーク形式に加えて、以下の形式も受け付けます：

  • デモ。 30分のチュートリアル形式によるツールのデモンストレーション。
  • 意見表明。 20分で OCaml の設計空間（機能の追加・削除など）に関するトピックを発表する形式。
  • 経験報告。 20分で OCaml またはツールの使用経験を報告する形式。

提案の方法

提案サイトはこちら：https://types-hotcrp.paris.inria.fr/ocaml26/

トークの説明（通常2〜3ページ程度。それより少なくても多くても構いません）を提出してください。内容は、取り上げる問題と提案する解決策・手法です。発表方法自体がトークのユニークな特徴だと思われる場合は、発表方法の説明も含めていただいて構いません。

LaTeX で作成した PDF が一般的ですが、必須ではありません。

昨年の採択発表はこちらでご覧いただけます：https://icfp25.sigplan.org/home/ocaml-2025#event-overview

  • 評価基準

    以下の基準で提案を評価します：

    • OCaml コミュニティへの関連性
    • 厳密さと健全性
    • 新規性
    • 明確さ
    • 魅力的かつ有益な発表ができる可能性

LLM 使用に関する推奨事項

LLM によって主に書かれた提案は受け付けられず、机上却下となります。文法の修正やスタイルの向上のための LLM 使用は全く問題ありません（特に英語が母国語でない場合）。しかし、内容を直接生成するための使用は危険かつ非専門的であり、著者性と査読者の努力を損なうものです。

開催地情報

今回の Caml Workshop は、8月末に開催される他のワークショップやイベントとともに Functional Programming Workshops (FPW) 2026 の一部としてパリで開催されます。これは通常の ICFP との共同開催から離れた形となります。FPW'26 の一部のイベントは ICFP とも共同開催されるため、ほぼ同時期に開催されます。

ML Workshop との連携

ML Family Workshop も今年 FPW'26 と共同開催されており、参加者にとって興味深い内容かもしれません。OCaml Workshop が OCaml エコシステムの開発に焦点を当てているのに対し、ML Workshop は言語に依存しない、より研究志向の内容となっています。提案が両方の場に適していると思われる著者は、提出時にその旨を示すか、プログラム委員長に連絡してください。
