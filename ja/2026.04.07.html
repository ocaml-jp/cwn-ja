restricted 2.0.0 プログラムがアクセスできるシステム操作とファイルシステムの範囲を制限する

Archive: https://discuss.ocaml.org/t/ann-restricted-2-0-0-limit-which-system-operations-and-which-parts-of-the-filesystem-your-program-can-access/17942/1

removewingman によるアナウンス

こんにちは、

restricted の新バージョンを発表できることを嬉しく思います。

このライブラリを使うと、プログラムがアクセスできるシステム操作とファイルシステムの範囲を制限できます。プログラムのできるだけ早い段階で呼び出すことで、以降のコードが権限を縮小した状態で実行されます。現在、実際に強制される制限が実装されているオペレーティングシステムは以下の通りです：

  • OpenBSD
  • Linux（ファイルシステムビューのみ）

他のオペレーティングシステムでも、restricted を使ってプログラムが必要とする権限を文書化することができます。ユーザーは pledge on Linux などのツールを使って、プログラムがこれらの約束を守っているかどうかをテストできます。ぜひご活用ください :slight_smile:

  • opam
  • api
  • examples
  • git repo
  • License AGPL

CHANGELOG:

  • テスト環境：
    • x86_64 openbsd（16/16 テスト合格）
    • x86_64-alpine-linux（Landlock ABI version: 7）（12テスト中4テスト合格：file_read_unv, file_write_unv, file_create_unv_linux, file_cwrr_unv_linux）

追加

  • API: exec の promises と accesses をオプションに変更
  • OpenBSD: exec promises

削除

  • SystemOperation.TmpPath（非推奨、pledge (2) を参照）

ocgtk 0.1: GTK 4 の OCaml バインディング（プレビューリリース）

Archive: https://discuss.ocaml.org/t/ann-ocgtk-0-1-ocaml-bindings-for-gtk-4-preview-release/17944/1

Chris Armstrong によるアナウンス

GTK 4 の OCaml バインディングである ocgtk の最初のプレビューリリース（v0.1-preview0）を発表できることを嬉しく思います。

ocgtk とは？

ocgtk は GTK 4 ウィジェットツールキットおよび関連ライブラリ（GDK、GIO、Pango、GdkPixbuf、GSK、Graphene、Cairo）の OCaml バインディングを提供します。現時点では OCaml >=5.3 を対象としています。

長年使われてきた GTK 3 バインディングライブラリである lablgtk3 にインスパイアされ、コアとなる共通コードや多くの設計思想を取り入れています。ただし API には互換性がありません。lablgtk3 のバインディングは主に手書きであるのに対し、ocgtk はコードジェネレーターを使用して GObject Introspection（GIR）定義ファイルを読み込み、C スタブと OCaml ラッパーの両方を生成します。これにより、バインディングのカバレッジは手書きの労力ではなく GIR 定義に追従し、生成されたコードは一貫した手続き型の構造に従います。

ツーリングに関する注記：コードジェネレーター自体は LLM の多大な支援を受けて開発されました。ただし、実際のライブラリを構成する C スタブと OCaml モジュールという出力は、LLM によって書かれたものではなく、GIR 定義からプログラム的に生成されたものです。

実装済みの機能

  • GTK 4、GDK、GIO、Pango、GdkPixbuf、GSK、Graphene、Cairo のバインディング
  • クラス、レコード、列挙型、ビットフィールド
  • コンストラクタ、メソッド、プロパティ
  • シグナル（戻り値やパラメーターなし）
  • 動作するサンプル：GTK 4 電卓アプリケーション

未実装の機能

これは最初のマイルストーンであり、本番対応リリースではありません。まだサポートされていない重要な機能がいくつかあります：

  • インターフェース：GObject インターフェースはまだサポートされていません。これは見た目以上に広範な影響があります。インターフェース型そのものが存在しないだけでなく、インターフェースをパラメーターとして受け取ったりインターフェース型を返したりするクラスのメソッドも除外されています。このため、それ以外はサポートされているクラスでも一部のメソッドがバインディングから欠落することがあります。
  • コールバック：シグナル以外の OCaml コールバックを引数として受け取る関数はまだサポートされていません。これにより、多くの非同期 GIO API や一部の GTK パターンの使用が制限されます。
  • フリー関数：クラスに属さないトップレベル関数はまだ生成されていません。そのため一部のユーティリティ API が利用できません。
  • 不透明でないレコードフィールドへのアクセス：不透明でない GLib/GDK レコードのフィールドはまだ直接読み取れません。
  • 複数パラメーターのシグナル：パラメーターがゼロで戻り値のないシグナルのみサポートされており、複雑なシグネチャを持つシグナルは暗黙的にスキップされます。
  • GLib 型：ほとんどの GLib 型はバインドされていないため、リスト、バイト配列、GVariant（DBus で多用される）を使用するいくつかの有用な API が利用できません。

これらの制限により、基本的な GTK アプリケーションは構築できますが、より高度なウィジェットの使用、非同期 I/O、またはインターフェースに大きく依存する API では壁にぶつかることが予想されます。

上記の多くに対応するための大まかな計画を記載した ROADMAP.md があります。

ドキュメント

API リファレンスドキュメントは現在不十分な状態です。GIR 定義からドキュメントを正確に生成することはこのプロジェクトにおいて未解決の問題であるため、現時点では生成されたモジュールのほとんどはドキュメント化されていません。当面は GTK 4 のドキュメントが各バインディングの対応を理解するための最良のリファレンスであり、リポジトリ内のサンプル（特に非自明な GTK 4 アプリケーションを示す電卓サンプル）が実際の使用方法を示しています。

安定性

このコードは本番環境でテストされていません。生成されたバインディングは広い範囲をカバーしていますが、テストカバレッジはまだ包括的ではありません。現段階では本番環境や安全性が重要なコンテキストでの使用はお勧めしません。コメントやマージリクエストは歓迎します。

このコードは Linux 環境以外ではテストされていません。

インストールと使用方法

現時点ではソースコードをダウンロードしてピンしてください（opam リリースの準備を進めていますが、クロスプラットフォームの問題の解決に時間がかかっています。利用可能になり次第、こちらのコメントで更新します）。

バインディングは GTK 4.14.5（Ubuntu 24.04）に対して生成されています。古いバージョンをお使いの場合、理論上はコンパイルできますが、お使いのバージョンで利用できない API を使用するとランタイムエラーが発生します。

  1. リポジトリをダウンロード：https://github.com/chris-armstrong/ocgtk%EF%BC%88%E3%81%93%E3%81%93%E3%81%A7%E8%A8%80%E5%8F%8A%E3%81%97%E3%81%9F%E5%86%85%E5%AE%B9%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6%E3%81%AF v0.1-preview0 タグを使用できます）
  2. 各サブディレクトリ内の conf-gtk4 および ocgtk パッケージをピン
    1. opam pin conf-gtk
    2. opam pin ocgtk
  3. opam install ocgtk を実行

各ライブラリは ocamlfind レベルでは ocgtk.<libname> として公開されています。例えば dune ファイルに ocgtk.gtk や ocgtk.gdk を追加し、opam / dune-project ファイルでは ocgtk に依存するようにしてください。

ソースとイシュートラッカー：https://github.com/chris-armstrong/ocgtk

フィードバックやバグ報告を歓迎します。

out-channel-redirect 0.1 — out_channel の出力をリダイレクト・キャプチャする

Archive: https://discuss.ocaml.org/t/ann-out-channel-redirect-0-1-redirect-and-capture-out-channel-output/17945/1

Hhugo によるアナウンス

out-channel-redirect の最初のリリースを発表できることを嬉しく思います。これは out_channel に書き込まれた出力をリダイレクト・キャプチャするためのライブラリです。

機能：

  • 任意の out_channel を文字列にキャプチャ（stdout と stderr の便利なショートハンドあり）
  • あるチャネルを別のチャネルにリダイレクト
  • ネイティブ（dup/dup2 経由）、JavaScript（js_of_ocaml）、WebAssembly（wasm_of_ocaml）で動作
  • リダイレクトのライフタイムを手動で制御するエキスパート API

インストール：

opam install out-channel-redirect

クイックサンプル：

let output, result =
  Out_channel_redirect.capture_channel some_channel ~f:(fun () → 
    Printf.fprintf some_channel "hello";
    42)
(* output = "hello", result = 42 *)

ppxlib.0.38.0

Archive: https://discuss.ocaml.org/t/ann-ppxlib-0-38-0/17948/1

Nathan Rebours によるアナウンス

ppxlib チームは ppxlib.0.38.0 のリリースを発表できることを嬉しく思います。

このリリースの主な機能は OCaml 5.5 のサポートと、OCaml 5.4 および 5.3 のサポートの改善です。また、いくつかのバグ修正と新しい API エントリも含まれています。詳細なリリースノートはこちらでご覧いただけます。

これはこのスレッドで新しいコンパイラサポートのアプローチを発表して以来の初リリースです。>= 5.3 のコンパイラバージョンに対するすべてのサポートはこの手法を使用して実装されており、新しい言語機能は AST にエンコードされています。ラベル付きタプルやエフェクトパターンなど、それらの一部を操作できるように Ast_builder と Ast_pattern 関数を追加しました。これらは API ドキュメントのこちらとこちらでご確認いただけます。また、新しいマニュアルセクションでそれらの使い方を例示しています。現在の内部 AST（5.2）では表現できない新しい言語機能の生成や消費に興味がある場合は、イシューを開いていただければ関連するヘルパーを追加します。

エコシステムが 5.2 の AST バンプと ppxlib.0.36.0 からまだ回復していない中で、新たな破壊的変更を導入することなく新しいコンパイラのサポートを追加し、ppxlib の範囲内でメンテナンスの負担を維持できることを嬉しく思います。 将来的にこのアプローチをさらに実用的にするために、コンパイラチームとの議論を開始しました。

このリリースで唯一の外部コントリビューターである @ceastlund に感謝します。

個人的には、このリリースの実現を助けてくれた @patricoferris、そして ppxlib での私の作業に資金提供してくださった OCaml Software Foundation と OCamlPro にも感謝します。

お読みいただきありがとうございます。良い前処理ライフを！

OCaml コードにおける AI 利用の自発的開示に関するプロポーザル

Archive: https://discuss.ocaml.org/t/a-proposal-for-voluntary-ai-disclosure-in-ocaml-code/17950/1

Anil Madhavapeddy によるアナウンス

皆さん、

opam メタデータとソースコード内の拡張属性を使用して、公開された OCaml コードにおける AI 利用を自発的に開示できるようにする ocaml-ai-disclosure プロポーザルをまとめました。

リポジトリとブログ投稿に詳細、属性を抽出するためのプロトタイプツール、および FAQ がありますが、簡単に言えば HTML に関する W3C 開示プロポーザルと非常に似たものを提案しています。

パッケージレベルの開示

opam パッケージは拡張フィールドを使用して開示を宣言できます： 

x-ai-disclosure: "ai-assisted"
x-ai-model: "claude-opus-4-6"
x-ai-provider: "Anthropic"

注：最終的なプロポーザルではただの値のリストになるかもしれませんが、概念はご理解いただけると思います。

OCaml モジュールレベル

OCaml は拡張属性をサポートしており、コンパイルユニット全体に適用されるフローティング属性を通じてこれを使用します： 

[@@@ai_disclosure "ai-generated"]
[@@@ai_model "claude-opus-4-6"]
[@@@ai_provider "Anthropic"]

let foo = ...
let bar = ...

単一のバインディングに適用される宣言属性を使用して、より細かくスコープを設定することもできます： 

[@@@ai_disclosure "ai-assisted"]

let human_written x = ...

let ai_helper y = ...
[@@ai_disclosure "ai-generated"]

開示は W3C HTML プロポーザルと同様の最近傍の祖先継承モデルに従い、明示的なアノテーションが継承された値を上書きします。

詳細はブログ投稿に記載されており、またプロポーザルリポジトリの FAQ にも一部の影響についての説明があります。

他の言語エコシステムで同様のことを試みた先行事例を見つけられなかったので、皆さんがご存じのものがあればぜひ教えてください。より広いエコシステムでこれへの関心がなければ、自分だけで使いますが、議論を始めても害はないと思い、提案してみました！

opentelemetry 0.90（および関連ライブラリ）

Archive: https://discuss.ocaml.org/t/ann-opentelemetry-0-90-and-associated-libraries/17958/1

Simon Cruanes によるアナウンス

こんにちは、

opentelemetry 0.90（および opentelemetry-client、opentelemetry-client-cohttp-lwt など）を発表できることを大変嬉しく思います。

Opentelemetry は https://opentelemetry.io/ の OCaml 実装で、ライブラリ等のインスツルメンテーションのためのコア クライアント API と、さまざまな エクスポーター を備えた完全な SDK の両方をカバーしています。Trace と互換性があります。

このリリースは数ヶ月の作業の成果であり、1.0 までの最後のリリースの一つになることが期待されています。変更履歴はかなり大きく、多くのことを見直しました。主な変更点は以下の通りです：

  • ocaml-protoc/pbrt 4.0 への移行（可変フィールドと protobuf の異なるコード生成）
  • API が https://opentelemetry.io のガイドラインに近づいた（まだ完全ではないが改善）
  • シグナルの内部処理のモジュール性が大幅に向上し、コードの再利用の改善、プラグイン可能なすべての機能、より良いキューイング/バッチング、HTTP リトライ
  • protobuf の代わりに JSON でエクスポートするサポート
  • trace 0.11 のサポート（これにより opentelemetry.trace はグローバルな OTEL エクスポーター以外のすべてのグローバル状態を排除できる）
  • ライフサイクルとクリーンアップの改善

API ドキュメントはこちらでご覧いただけます。いつものように、フィードバックやバグ報告などを大歓迎します。
