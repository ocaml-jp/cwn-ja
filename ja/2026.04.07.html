restricted 2.0.0 プログラムがアクセスできるシステム操作とファイルシステムの範囲を制限する

Archive: https://discuss.ocaml.org/t/ann-restricted-2-0-0-limit-which-system-operations-and-which-parts-of-the-filesystem-your-program-can-access/17942/1

removewingman によるアナウンス

こんにちは、

restricted の新バージョンをアナウンスできることを嬉しく思います。

このライブラリは、プログラムがアクセスできるシステム操作とファイルシステムの範囲を制限するためのものです。プログラムのできるだけ早い段階で呼び出すことで、残りのコードが権限を制限した状態で実行されるようになります。現在、実際に強制される制限が実装されているオペレーティングシステムは以下の通りです：

  • OpenBSD
  • Linux（ファイルシステムビューのみ）

他のオペレーティングシステムでも、restricted を使用してプログラムが必要とする権限を文書化することができます。ユーザーはpledge on Linux などのツールを使って、プログラムがこれらの約束を遵守しているかテストできます。どうぞお楽しみください :slight_smile:

  • opam
  • api
  • examples
  • git repo
  • License AGPL

変更履歴：

  • テスト済み環境：
    • x86_64 openbsd（16/16 テスト通過）
    • x86_64-alpine-linux（Landlock ABI version: 7）（12件中4件通過：file_read_unv、file_write_unv、file_create_unv_linux、file_cwrr_unv_linux）

追加

  • API: exec の promises と accesses をオプション化
  • OpenBSD: exec promises

削除

  • SystemOperation.TmpPath（非推奨、pledge (2) を参照）

ocgtk 0.1：GTK 4 向け OCaml バインディング（プレビューリリース）

Archive: https://discuss.ocaml.org/t/ann-ocgtk-0-1-ocaml-bindings-for-gtk-4-preview-release/17944/1

Chris Armstrong によるアナウンス

GTK 4 向け OCaml バインディングである ocgtk の最初のプレビューリリース（v0.1-preview0）をアナウンスできることを嬉しく思います。

ocgtk とは？

ocgtk は GTK 4 ウィジェットツールキットおよびその関連ライブラリ（GDK、GIO、Pango、GdkPixbuf、GSK、Graphene、Cairo）向けの OCaml バインディングです。現時点では OCaml >=5.3 を対象としています。

lablgtk3（長年にわたる GTK 3 バインディングライブラリ）からインスピレーションを受けており、コアとなる共通コードや多くの設計アイデアを取り入れています。ただし、APIには互換性がありません。lablgtk3 のバインディングが大部分を手書きで書かれているのに対し、ocgtk はコードジェネレーターを使用して GObject Introspection（GIR）定義ファイルを読み込み、C スタブと OCaml ラッパーの両方を生成します。これにより、バインディングのカバレッジは手書きの作業ではなく GIR 定義に追随し、生成されたコードは一貫した手続き的な構造に従います。

ツールについての注記：コードジェネレーター自体はLLMの多大な支援を受けて開発されました。ただし、実際のライブラリを形成するCスタブとOCamlモジュール（出力）は、LLMによって書かれたものではなく、GIR定義からプログラム的に生成されています。

実装済みの機能

  • GTK 4、GDK、GIO、Pango、GdkPixbuf、GSK、Graphene、Cairo のバインディング
  • クラス、レコード、列挙型、ビットフィールド
  • コンストラクタ、メソッド、プロパティ
  • シグナル（戻り値やパラメータなし）
  • 動作するサンプル：GTK 4 計算機アプリケーション

不足している機能

これは最初のマイルストーンであり、本番環境向けのリリースではありません。いくつかの重要な機能はまだサポートされていません：

  • インターフェース：GObject インターフェースはまだサポートされていません。これは見た目よりも広範な影響があります。インターフェース型自体が存在しないだけでなく、パラメータとしてインターフェースを受け取ったりインターフェース型を返したりするクラスのメソッドも抑制されます。つまり、それ以外はサポートされているクラスの一部のメソッドがバインディングから欠落します。
  • コールバック：シグナル以外の OCaml コールバックを引数として受け取る関数はまだサポートされていません。これにより、多くの非同期 GIO API や一部の GTK パターンの使用が制限されます。
  • フリー関数：クラスに属さないトップレベル関数はまだ生成されていません。そのため、一部のユーティリティ API が利用できません。
  • 非オペーク型レコードフィールドへのアクセス：非オペーク型の GLib/GDK レコードのフィールドを直接読み取ることはまだできません。
  • 複数パラメータを持つシグナル：パラメータがなく戻り値もないシグナルのみサポートされています。複雑なシグネチャを持つシグナルは静かにスキップされます。
  • GLib 型：ほとんどの GLib 型はバインドされていないため、リスト、バイト配列、GVariant（DBus で多用される）を使用するいくつかの有用な API が利用できません。

これらの制限により、基本的な GTK アプリケーションは作成できますが、より高度なウィジェットの使用、非同期 I/O、またはインターフェースに大きく依存する API では行き詰まることが予想されます。

上記の多くに対処するための大まかな計画を記載した ROADMAP.md があります。

ドキュメント

API リファレンスドキュメントは現在不十分です。GIR 定義から正確にドキュメントを生成することはこのプロジェクトにとって未解決の問題であるため、現時点では生成されたモジュールはほとんどドキュメント化されていません。それまでの間、各バインディングに対応するものを理解するための最良のリファレンスは GTK 4 ドキュメントであり、リポジトリ内のサンプル（特に、非自明な GTK 4 アプリケーションを実演する計算機のサンプル）は実際の使用方法を示しています。

安定性

このコードは本番環境でテストされていません。生成されたバインディングは広い範囲をカバーしていますが、テストカバレッジはまだ十分ではありません。現時点では本番環境や安全性が重要な状況での使用はお勧めしません。コメントや MR は歓迎します。

このコードは Linux 環境以外ではテストされていません。

インストールと使用方法

現時点ではソースコードをダウンロードして pin してください（opam リリースの準備中です。クロスプラットフォームの問題を整理するのに時間がかかっているため、準備ができ次第コメントで更新します）。

バインディングは GTK 4.14.5（Ubuntu 24.04）に対して生成されています。古いバージョンをお持ちの場合、理論上はコンパイルできますが、お使いのバージョンで利用できない API についてはランタイムエラーが発生します。

  1. リポジトリをダウンロード：https://github.com/chris-armstrong/ocgtk%EF%BC%88%E3%81%93%E3%81%93%E3%81%A7%E8%A8%80%E5%8F%8A%E3%81%97%E3%81%9F%E5%86%85%E5%AE%B9%E3%82%92%E5%8F%96%E5%BE%97%E3%81%99%E3%82%8B%E3%81%AB%E3%81%AF v0.1-preview0 タグを使用できます）
  2. それぞれのサブディレクトリにある conf-gtk4 と ocgtk パッケージを pin する
    1. opam pin conf-gtk
    2. opam pin ocgtk
  3. opam install ocgtk を実行する

各ライブラリは ocamlfind レベルで ocgtk.<libname> として公開されています。例えば、dune ファイルに ocgtk.gtk や ocgtk.gdk を追加し、opam / dune-project ファイルでは ocgtk に依存します。

ソースとイシュートラッカー：https://github.com/chris-armstrong/ocgtk

フィードバックやバグ報告を歓迎します。

out-channel-redirect 0.1 — out_channel の出力のリダイレクトとキャプチャ

Archive: https://discuss.ocaml.org/t/ann-out-channel-redirect-0-1-redirect-and-capture-out-channel-output/17945/1

Hhugo によるアナウンス

out-channel-redirect の最初のリリースをアナウンスできることを嬉しく思います。これは out_channel に書き込まれた出力をリダイレクトおよびキャプチャするためのライブラリです。

機能：

  • 任意の out_channel を文字列にキャプチャ（stdout と stderr 用の便利な短縮形あり）
  • あるチャンネルを別のチャンネルにリダイレクト
  • ネイティブ（dup/dup2 経由）、JavaScript（js_of_ocaml）、WebAssembly（wasm_of_ocaml）で動作
  • 手動でリダイレクトのライフタイムを制御するためのエキスパート API

インストール：

opam install out-channel-redirect

クイックサンプル：

let output, result =
  Out_channel_redirect.capture_channel some_channel ~f:(fun () → 
    Printf.fprintf some_channel "hello";
    42)
(* output = "hello", result = 42 *)

ppxlib.0.38.0

Archive: https://discuss.ocaml.org/t/ann-ppxlib-0-38-0/17948/1

Nathan Rebours によるアナウンス

ppxlib チームは ppxlib.0.38.0 のリリースをアナウンスできることを嬉しく思います。

このリリースの主な特徴は OCaml 5.5 のサポートと、OCaml 5.4 および 5.3 のサポート改善です。また、いくつかのバグ修正と新しい API エントリも含まれています。詳細なリリースノートはこちらでご覧いただけます。

これはこのスレッドで新しいコンパイラサポートのアプローチを発表してから初めてのリリースです。>= 5.3 のコンパイラバージョンへのサポートはすべてこの技術を使用して実装されており、新しい言語機能は AST にエンコードされています。ラベル付きタプルやエフェクトパターンなど、それらの一部を操作できるように Ast_builder と Ast_pattern の関数を追加しました。これらは API ドキュメントのこちらとこちらでご覧いただけます。また、新しいマニュアルセクションでその使用方法も説明しています。現在の内部 AST（5.2）では表現できない他の新しい言語機能の生成または消費に興味がある場合は、関連するヘルパーを追加できるようイシューを開いてください。

エコシステムがまだ 5.2 AST の変更と ppxlib.0.36.0 から回復しきれていない状況で、さらなる破壊的変更を導入することなく新しいコンパイラへのサポートを追加し、メンテナンスの負担を ppxlib の範囲内に保てることを嬉しく思います。 私たちはこのアプローチを将来的にさらに実行可能にするために、コンパイラチームとの議論を開始しました。

このリリースの唯一の外部コントリビューターである @ceastlund に感謝します。

個人的には、リリースを実現するために助けてくださった @patricoferris、そして ppxlib での私の作業に資金を提供してくださった OCaml Software Foundation と OCamlPro にも感謝します。

お読みいただきありがとうございます。Happy preprocessing!

OCaml コードにおける AI 開示の自発的な取り組みに関する提案

Archive: https://discuss.ocaml.org/t/a-proposal-for-voluntary-ai-disclosure-in-ocaml-code/17950/1

Anil Madhavapeddy によるアナウンス

皆さん、

opam メタデータと拡張属性を使用して、公開された OCaml コードにおける AI 使用状況を自発的に開示できるようにする ocaml-ai-disclosure 提案をまとめました。

リポジトリとブログ記事に詳細、属性を抽出するためのプロトタイプツール、および FAQ がありますが、要約すると、HTML 向けの W3C 開示提案と非常に似たものを提案しています。

パッケージレベルの開示

opam パッケージは拡張フィールドを使用して開示を宣言できます： 

x-ai-disclosure: "ai-assisted"
x-ai-model: "claude-opus-4-6"
x-ai-provider: "Anthropic"

注：最終的な提案では単純な値のリストになるかもしれませんが、考え方はこのようなものです。

OCaml モジュールレベル

OCaml は拡張属性をサポートしており、コンパイル単位全体に適用されるフローティング属性を介して使用します： 

[@@@ai_disclosure "ai-generated"]
[@@@ai_model "claude-opus-4-6"]
[@@@ai_provider "Anthropic"]

let foo = ...
let bar = ...

これらは、単一のバインディングに適用される宣言属性を介してより細かいスコープで指定することもできます： 

[@@@ai_disclosure "ai-assisted"]

let human_written x = ...

let ai_helper y = ...
[@@ai_disclosure "ai-generated"]

開示は W3C HTML 提案と同様の最近祖先継承モデルに従い、明示的なアノテーションが継承された値を上書きします。

詳細についてはブログ記事と、いくつかの意味合いについての提案リポジトリ内の FAQをご覧ください。

他の言語エコシステムが同様の取り組みをしている先行事例を見つけられなかったので、皆さんがご存知のものがあればぜひ教えてください。より広いエコシステムでこれに対する関心がなければ、自分自身で使うだけにしますが、議論を始めることに問題はないと思いました！

opentelemetry 0.90（および関連ライブラリ）

Archive: https://discuss.ocaml.org/t/ann-opentelemetry-0-90-and-associated-libraries/17958/1

Simon Cruanes によるアナウンス

こんにちは、

opentelemetry 0.90（および opentelemetry-client、opentelemetry-client-cohttp-lwt など）をアナウンスできることを嬉しく思います。

opentelemetry は OCaml 向けの https://opentelemetry.io/ の実装です。ライブラリなどのインストルメンテーション用のコア クライアント API と、様々な エクスポーター を備えたフル SDK の両方をカバーしています。Trace と互換性があります。

このリリースは数ヶ月の作業の成果であり、1.0 前の最後のリリースの一つになる見込みです。変更履歴はかなり大きく、多くのことを見直しました。主な変更点は以下の通りです：

  • ocaml-protoc/pbrt 4.0 への移行（ミュータブルフィールドと異なるコードジェン）
  • API が https://opentelemetry.io のガイドラインに近くなった（完全ではないがより改善）
  • シグナルの内部処理のモジュール性が大幅に改善され、コードの再利用性向上、すべてのものをプラグイン可能に、キューイング/バッチ処理の改善、HTTP リトライのサポート
  • プロトコルバッファの代わりに JSON としてエクスポートするサポート
  • trace 0.11 のサポート（opentelemetry.trace がグローバル OTEL エクスポーター以外のグローバル状態を持たないようにできる）
  • ライフサイクルとクリーンアップの改善

API ドキュメントはこちらでご覧いただけます。いつものように、フィードバック、バグ報告などは大歓迎です。
