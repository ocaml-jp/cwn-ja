* restricted 2.0.0 プログラムがアクセスできるシステム操作とファイルシステムの範囲を制限する
:PROPERTIES:
:CUSTOM_ID: 1
:END:
Archive: https://discuss.ocaml.org/t/ann-restricted-2-0-0-limit-which-system-operations-and-which-parts-of-the-filesystem-your-program-can-access/17942/1

** removewingman さんのアナウンス


こんにちは、

[[https://opam.ocaml.org/packages/restricted/][restricted]] の新バージョンをリリースしましたことをお知らせします。

このライブラリを使うと、プログラムがアクセスできるシステム操作とファイルシステムの範囲を制限することができます。プログラムの早い段階で呼び出すことで、その後のコードを縮小された権限で実行させることができます。現在、実際の制限の強制は以下のオペレーティングシステムで実装されています：

- OpenBSD
- Linux（ファイルシステムビューのみ）

他のオペレーティングシステムでも、[[https://removewingman.codeberg.page/restricted/][restricted]] を使ってプログラムが必要とする権限を文書化することができます。その後、[[https://justine.lol/pledge/][pledge on Linux]] などのツールを使って、プログラムがこれらの約束を守っているかをユーザーがテストできます。お楽しみください :slight_smile:

- [[https://opam.ocaml.org/packages/restricted/restricted.1.2.0/][opam]]
- [[https://removewingman.codeberg.page/restricted/][api]]
- [[https://codeberg.org/removewingman/restricted/src/branch/main#usage-and-examples][examples]]
- [[https://codeberg.org/removewingman/restricted][git repo]]
- [[https://en.wikipedia.org/wiki/GNU_Affero_General_Public_License][License AGPL]]

変更履歴：

- テスト環境：
  * x86_64 openbsd（16/16テスト合格）
  * x86_64-alpine-linux（Landlock ABI バージョン：7）（12テスト中4テスト合格：file_read_unv、file_write_unv、file_create_unv_linux、file_cwrr_unv_linux）

*** 追加

- API：exec promises と accesses をオプション化
- OpenBSD：exec promises

*** 削除

- SystemOperation.TmpPath（非推奨、~pledge (2)~ を参照）

* ocgtk 0.1：GTK 4 向け OCaml バインディング（プレビューリリース）
:PROPERTIES:
:CUSTOM_ID: 3
:END:
Archive: https://discuss.ocaml.org/t/ann-ocgtk-0-1-ocaml-bindings-for-gtk-4-preview-release/17944/1

** Chris Armstrong さんのアナウンス


GTK 4 向け OCaml バインディングである ocgtk の最初のプレビューリリース（v0.1-preview0）を発表できることを嬉しく思います。

*** ocgtk とは？

[[https://github.com/chris-armstrong/ocgtk][ocgtk]] は GTK 4 ウィジェットツールキットおよびその関連ライブラリ（GDK、GIO、Pango、GdkPixbuf、GSK、Graphene、Cairo）向けの OCaml バインディングを提供します。現在のところ OCaml >=5.3 を対象としています。

長年使われてきた GTK 3 バインディングライブラリである lablgtk3 から着想を得ており、コアとなる共通コードや多くの設計思想を取り入れています。ただし、API には互換性がありません。lablgtk3 のバインディングが大部分を手書きで記述しているのに対し、ocgtk は GObject Introspection（GIR）定義ファイルを読み込んで C スタブと OCaml ラッパーの両方を生成するコードジェネレーターを使用しています。これにより、バインディングのカバレッジは手書きの作業量ではなく GIR 定義に追従し、生成されたコードは一貫した手続き的な構造に従います。

*ツールについての注記*：コードジェネレーター自体は LLM の多大な支援を受けて開発されました。ただし、実際のライブラリを構成する C スタブと OCaml モジュール（出力）は GIR 定義からプログラム的に生成されたものであり、LLM によって書かれたものではありません。

*** 実装済みの機能

- GTK 4、GDK、GIO、Pango、GdkPixbuf、GSK、Graphene、Cairo のバインディング
- クラス、レコード、列挙型、ビットフィールド
- コンストラクター、メソッド、プロパティ
- シグナル（戻り値やパラメーターなし）
- 動作するサンプル：GTK 4 の電卓アプリケーション

*** 未実装の機能

これは最初のマイルストーンであり、プロダクションレディなリリースではありません。いくつかの重要な機能はまだサポートされていません：

- *インターフェイス*：GObject インターフェイスはまだサポートされていません。これは見た目以上に広範な影響があります。インターフェイス型自体が存在しないだけでなく、パラメーターとしてインターフェイスを受け取ったりインターフェイス型を返すクラスのメソッドも抑制されます。そのため、それ以外はサポートされているクラスのいくつかのメソッドがバインディングから欠落することになります。
- *コールバック*：シグナル以外で OCaml コールバックを引数として受け取る関数はまだサポートされていません。これにより、多くの非同期 GIO API や一部の GTK パターンの使用が制限されます。
- *フリー関数*：クラスに属さないトップレベル関数はまだ生成されません。そのため、一部のユーティリティ API は利用できません。
- *非オペーク型レコードフィールドへのアクセス*：非オペーク型の GLib/GDK レコードのフィールドをまだ直接読み取ることができません。
- *複数パラメーターのシグナル*：パラメーターが0個で戻り値なしのシグナルのみサポートされており、複雑なシグナルは静かにスキップされます。
- *GLib 型*：ほとんどの GLib 型はバインドされていないため、リスト、バイト配列、GVariant（DBus で多用される）を使用する多くの有用な API が利用できません。

これらの制限により、基本的な GTK アプリケーションは構築できますが、より高度なウィジェットの使用、非同期 I/O、またはインターフェイスに大きく依存する API では壁に当たることを覚悟してください。

上記の多くに対処するおおまかな計画をまとめた [[https://github.com/chris-armstrong/ocgtk/blob/main/ROADMAP.md][ROADMAP.md]] があります。

*** ドキュメント

API リファレンスドキュメントは現在不十分な状態です。GIR 定義からドキュメントを正確に生成することはこのプロジェクトの未解決の課題であるため、現時点では生成されたモジュールはほとんどドキュメントがありません。当面は、各バインディングが何に対応するかを理解するための最良のリファレンスとして GTK 4 のドキュメントをご参照ください。また、リポジトリ内のサンプル（特に非自明な GTK 4 アプリケーションを示す電卓サンプル）が実践的な使い方を示しています。

*** 安定性

このコードはプロダクション環境でのテストが行われていません。生成されたバインディングは広い面積をカバーしていますが、テストカバレッジはまだ包括的ではありません。この段階では、プロダクション環境や安全性が重要な用途での使用はお控えください。コメントや MR は歓迎します。

このコードは Linux 環境以外ではテストされていません。

*** インストールと使い方

現在はソースコードをダウンロードしてピンしてください（opam リリースの準備中です。クロスプラットフォームの問題の整理に多少時間がかかっていますので、準備ができ次第コメントで更新します）。

バインディングは GTK 4.14.5（Ubuntu 24.04）に対して生成されています。古いバージョンをお持ちの場合でも（理論上は）コンパイルできますが、お使いのバージョンで利用できない API についてはランタイムエラーが発生します。

1. リポジトリをダウンロードする：https://github.com/chris-armstrong/ocgtk（ここで述べた内容を取得するには v0.1-preview0 タグを使用できます）
2. それぞれのサブディレクトリにある conf-gtk4 と ocgtk パッケージをピンする
   1. ~opam pin conf-gtk~
   2. ~opam pin ocgtk~
3. ~opam install ocgtk~ を実行する

各ライブラリは ocamlfind レベルで ~ocgtk.<libname>~ として公開されています。例えば、dune ファイルに ~ocgtk.gtk~ や ~ocgtk.gdk~ を追加し、~opam~ / ~dune-project~ ファイルでは ~ocgtk~ に依存します。

*ソースとイシュートラッカー*：https://github.com/chris-armstrong/ocgtk

フィードバックやバグ報告を歓迎します。
      



* out-channel-redirect 0.1 — out_channel の出力をリダイレクト・キャプチャする
:PROPERTIES:

Archive: https://discuss.ocaml.org/t/ann-out-channel-redirect-0-1-redirect-and-capture-out-channel-output/17945/1

** Hhugo さんのアナウンス


[[https://github.com/hhugo/out-channel-redirect][out-channel-redirect]] の最初のリリースをお知らせします。これは out_channel に書き込まれた出力をリダイレクトおよびキャプチャするためのライブラリです。

*** 機能：

- 任意の out_channel を文字列にキャプチャ（stdout と stderr 用の便利なショートハンド付き）
- あるチャネルを別のチャネルにリダイレクト
- ネイティブ（dup/dup2 経由）、JavaScript（js_of_ocaml）、WebAssembly（wasm_of_ocaml）で動作
- 手動リダイレクトライフタイム制御のためのエキスパート API

*** インストール：
#+begin_example
opam install out-channel-redirect
#+end_example

*** クイックサンプル：
#+begin_src ocaml
let output, result =
  Out_channel_redirect.capture_channel some_channel ~f:(fun () → 
    Printf.fprintf some_channel "hello";
    42)
(* output = "hello", result = 42 *)
#+end_src
      



* ppxlib.0.38.0

:CUSTOM_ID: 5

** Nathan Rebours さんのアナウンス


ppxlib チームは ppxlib.0.38.0 のリリースをお知らせします。

このリリースの主な機能は OCaml 5.5 のサポートと、OCaml 5.4 および 5.3 のサポート改善です。いくつかのバグ修正と新しい API エントリも含まれており、詳細なリリースノートは[[https://github.com/ocaml-ppx/ppxlib/releases/tag/0.38.0][こちら]]でお読みいただけます。

これは、[[https://discuss.ocaml.org/t/ann-ppxlib-support-for-future-compilers/17430][このスレッド]]で新しいコンパイラサポートへの新しいアプローチを発表して以来の最初のリリースです。~>= 5.3~ のコンパイラバージョンへのサポートはすべてこの手法を使って実装されており、新しい言語機能は AST にエンコードされています。ラベル付きタプルやエフェクトパターンなど、それらの一部を操作できるよう ~Ast_builder~ と ~Ast_pattern~ の関数を追加しました。これらは API ドキュメントの[[https://ocaml-ppx.github.io/ppxlib/ppxlib/Ppxlib/Ast_builder/Default/index.html#future-asts][こちら]]と[[https://ocaml-ppx.github.io/ppxlib/ppxlib/Ppxlib/Ast_pattern/index.html#future-asts][こちら]]でご覧いただけます。また、[[https://ocaml-ppx.github.io/ppxlib/ppxlib/compatibility.html#future_compilers][マニュアルの新しいセクション]]でその使い方を例示しています。現在の内部 AST（5.2）では表現できない他の新しい言語機能の生成や消費に興味がある場合は、[[https://github.com/ocaml-ppx/ppxlib/issues/new][イシューを開いて]]ください。関連するヘルパーを追加します。

エコシステムがまだ 5.2 AST の変更と ~ppxlib.0.36.0~ から回復していない中、新しいコンパイラへのサポートをさらなる破壊なく追加し、メンテナンスの負担を ppxlib の範囲内に収められることを嬉しく思います。
将来的にこのアプローチをさらに実現可能なものにするために、コンパイラチームとの[[https://github.com/ocaml/ocaml/issues/14668][議論]]を開始しました。

このリリースで唯一の外部コントリビューターである @ceastlund に感謝します。

リリースを助けてくれた @patricoferris、そして ppxlib での私の作業に資金提供してくださった OCaml Software Foundation と OCamlPro にも個人的に感謝します。

お読みいただきありがとうございます。良いプリプロセッシングを！
      



* OCaml コードにおける AI 開示の自発的な提案
:PROPERTIES:
:CUSTOM_ID: 6
:END:
Archive: https://discuss.ocaml.org/t/a-proposal-for-voluntary-ai-disclosure-in-ocaml-code/17950/1

** Anil Madhavapeddy さんのアナウンス


皆さん、

opam メタデータと[[https://ocaml.org/manual/5.3/attributes.html][拡張属性]]を使って、公開された OCaml コードにおける AI 使用を自発的に開示するための *[[https://github.com/avsm/ocaml-ai-disclosure][ocaml-ai-disclosure]]* 提案をまとめました。

リポジトリと[[https://anil.recoil.org/notes/opam-ai-disclosure][ブログ記事]]には詳細、属性を抽出するためのプロトタイプツール、FAQ が含まれていますが、一言で言えば [[https://www.w3.org/community/ai-content-disclosure/][HTML に関する W3C 開示提案]]と非常に似たものを提案しています。

*** パッケージレベルの開示

opam パッケージは拡張フィールドを使って開示を宣言できます：

#+begin_example
x-ai-disclosure: "ai-assisted"
x-ai-model: "claude-opus-4-6"
x-ai-provider: "Anthropic"
#+end_example

/注：最終的な提案では単純に値のリストになる可能性がありますが、概念はご理解いただけると思います。/

*** OCaml モジュールレベル

OCaml は拡張属性をサポートしており、コンパイルユニット全体に適用されるフローティング属性を通じてこれを使用します：

#+begin_src ocaml
[@@@ai_disclosure "ai-generated"]
[@@@ai_model "claude-opus-4-6"]
[@@@ai_provider "Anthropic"]

let foo = ...
let bar = ...
#+end_src

これらは単一のバインディングに適用される宣言属性を通じてより細かくスコープを絞ることもできます：

#+begin_src ocaml
[@@@ai_disclosure "ai-assisted"]

let human_written x = ...

let ai_helper y = ...
[@@ai_disclosure "ai-generated"]
#+end_src

開示は W3C HTML 提案と同様の最近の祖先継承モデルに従い、明示的なアノテーションが継承された値を上書きします。

詳細については[[https://anil.recoil.org/notes/opam-ai-disclosure][ブログ記事]]を書きました。また、いくつかの含意について[[https://github.com/avsm/ocaml-ai-disclosure][提案リポジトリの FAQ]] もあります。

他の言語エコシステムで同様のことを試みた事例は見つけられなかったので、皆さんがご存知の事例があればぜひ教えてください。より広いエコシステムでこれに関心がなければ自分だけで使うつもりですが、議論を始めることに害はないと思いました！

* opentelemetry 0.90（および関連ライブラリ）
:PROPERTIES:
:CUSTOM_ID: 9
:END:
Archive: https://discuss.ocaml.org/t/ann-opentelemetry-0-90-and-associated-libraries/17958/1

** Simon Cruanes さんのアナウンス


こんにちは、

[[https://github.com/ocaml-tracing/ocaml-opentelemetry/releases/tag/v0.90][opentelemetry 0.90]]（および ~opentelemetry-client~、~opentelemetry-client-cohttp-lwt~ など）のリリースをお知らせします。

Opentelemetry は OCaml 向けの https://opentelemetry.io/ の実装です。コアの _クライアント API_（ライブラリ等のインストルメンテーション用）と、さまざまな _エクスポーター_ を備えた完全な SDK の両方をカバーしています。[[https://github.com/ocaml-tracing/ocaml-trace/][Trace]] と互換性があります。

このリリースは数か月の作業の成果であり、1.0 に向けた最後のリリースの一つになることを願っています。[[https://github.com/ocaml-tracing/ocaml-opentelemetry/blob/main/CHANGES.md#090][変更履歴]]はかなり大きく、多くのことを見直しました。主な変更点は以下の通りです：
- ocaml-protoc/pbrt 4.0 への移行（mutable フィールドと protobuf の異なるコード生成）
- API が https://opentelemetry.io のガイドラインに近づいた（まだ完全ではありませんが改善）
- シグナルの内部処理のモジュール性が大幅に改善され、コードの再利用が向上し、すべてのコンポーネントがプラガブルになり、キューイング/バッチング、HTTP リトライが改善
- protobuf の代わりに JSON としてエクスポートするサポート
- trace 0.11 のサポート（~opentelemetry.trace~ がグローバルな OTEL エクスポーター以外のすべてのグローバル状態を省略できるようになった）
- ライフサイクルとクリーンアップの改善
