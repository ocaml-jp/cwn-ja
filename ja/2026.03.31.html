Merlinのsignature-help機能の改善

アーカイブ: https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1

Charlène_Grosによるアナウンス

Taridesでは、Merlinのsignature-help機能に対する改善をお知らせできることを大変嬉しく思います!

念のため説明すると、signature-helpはMerlinの機能で、関数のシグネチャを表示し、アクティブなパラメータをハイライトします。エディタでは、通常、ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。例えば、次の画像はVSCodeでの関数exec_progのsignature-helpを表示しています。

https://us1.discourse-cdn.com/flex020/uploads/ocaml/original/2X/4/46e99c5c69bab991b8dd6a6a7494db5c2f32e07b.png

signature-helpコマンドの実装にはいくつかの欠点がありましたが、それらが修正されました。これらの修正により、この機能がはるかに使いやすくなっています。これらの改善のいくつかは、最新のMerlinとocaml-lspのリリースにすでに含まれており、その他は次回のリリースに含まれる予定です。

結果:

  • Merlinは関数パラメータに対してのみsignature-help情報を表示し、関数名には表示しなくなりました。
  • signature-helpはアクティブなパラメータに対してのみ関数の情報を表示し、最後のパラメータの後に最初のパラメータをハイライトするようにループバックしなくなりました。
  • signature-helpは、関数呼び出しの後にinがまだ書かれていないlet .. inバインディング内で使用されても、トリガーされるようになりました
  • オプショナルパラメータは、アクティブなパラメータがオプショナルパラメータの記述を開始したときに、検出されてハイライトされるようになりました。

新しいセクション:

  • AI/LLMの使用(4問) - OCaml開発にAI/LLMツールを使用していますか? どれを使用していますか? 何のために?
  • デバッグとプロファイリング(3問) - どのようなデバッグアプローチとツールを使用していますか? 最大の課題は何ですか? (2023年の自由記述回答でデバッグの困難さが頻繁に言及されたことが動機)

  • コミュニティサイズの好み(「コミュニティが…であることを望む」)

  • ツールとパッケージリポジトリのフィードバック用の専用自由記述フィールド

削除された質問(合計10問) - 実用性の低いデータを生み出した質問や冗長な質問、含まれるもの: 最大プロジェクトサイズ、リリーススケジュールの満足度、ソースホスティングプラットフォーム、Webデプロイメントプラットフォーム、「OCamlツールは快適なワークフローを提供する」、「OCamlライブラリは十分安定している」、「OCamlで書かれたソフトウェアは保守しやすい」、および言語機能のウィッシュリスト(エフェクトハンドラーはOCaml 5に搭載済み)。

変更された質問 - エコシステムの進化を反映した新しい回答オプション: OxCaml、Zig、Helix、Zed、WebAssemblyターゲット、duneパッケージ管理、Nixベースのビルド、Bluesky、および「切望する機能」セクションのAI/LLM関連オプション。古いコンパイラバージョン(4.02-4.06)は「≤4.07」に統合。ベンチマークツールはカテゴリーではなく特定の名前でリストアップされるようになりました。

ortac-0.8 Domainsを使った仕様駆動テスト

アーカイブ: https://discuss.ocaml.org/t/ann-ortac-0.8-specification-driven-testing-with-domains/17927/1

Nicolas Osborneによるアナウンス

こんにちは! Taridesでは、仕様駆動テストのためのortac-0.8.0のリリースを発表できることを大変嬉しく思います!

ortacはGospelエコシステムのツールです。中心となるアイデアは、Gospel仕様言語のサブセットをOCamlコードに変換し、これらの変換を使用してランタイムチェックを生成することです。

ortacはプラグインアーキテクチャを採用しています。opam経由でインストールできます:

$ opam install ortac-wrapper    # ランタイムアサーションチェック用のwrapperプラグインを、必要な他のすべての部品(ランタイムとcli)とともにインストール
$ opam install ortac-qcheck-stm # QCheck-STMテストを生成するqcheck-stmプラグインを、必要な他のすべての部品(ランタイムとcli)とともにインストール
$ opam install ortac-dune       # 前述の2つのプラグインを使用するためのduneボイラープレートを生成するduneプラグインをインストール

このリリースは、Ortac/QCheck-STMがQCheck-STMテストフレームワークのより多くの機能を活用できるようにすることに焦点を当てています。具体的には、並列コンテキストでのテストとコマンド生成の柔軟性です。

動作の仕組みを説明するドキュメントをご覧いただけます。

並列コンテキストでのテストについては、バージョン0.2でのバグレポート機能の導入と、バージョン0.4でのSUTを返す関数のカバレッジにより、並列コンテキストでQCheck-STMテストを簡単に生成することができなくなっていました(手書きのQCheck-STMテストでは非常に簡単です)。これが修正されました!

バグレポートの情報は、実際のテストを行う関数(実際の実行結果とモデルからの結果を比較する)で部分的に収集されます。つまり、並列テスト用のバグレポート機能を実装するには、この関数を書き直す必要があります。この関数はQCheck-STM+Domainsの中核であり、テストを実行する際に信頼するもの(いわばTrusted Testing Base)です。同じ動作を維持すること、生成されたテストが対応する手書きのQCheck-STMテストと同じセマンティクスを持つことを確認するために、情報の収集を導入するコミットのdiffを最小化し、ロジックの保持が明らかになるようにしました(結局のところ、私たちが信頼するのは人間によるレビューです)。

バージョン0.4以降、SUTを返す関数がテストに含まれています。新しく作成されたSUTは、次の呼び出しの引数として選ばれるSUTのストアに追加されます。ここでの問題は、並列コンテキストでそれらをどうするかです。並列ブランチで作成されたSUTをグローバルストアに追加することはできません。なぜなら、それはドメイン間で共有されるべきではないからです。シンプルな設計を選択し、並列コンテキストに入ったら新しく作成されたSUTの保存を停止します。ご心配なく、これらのSUTを返す関数は、テストのシーケンシャル部分(シーケンシャルモードおよび/または並列モードのシーケンシャルプレフィックス)で完全にテストされます。

QCheck-STMテストフレームワークの強力な機能の1つは、コマンドジェネレーターの柔軟性です。この柔軟性は、QCheck.Gen API自体と、QCheck-STMコマンドジェネレーターが現在のモデルの状態に対してパラメータ化されているという事実から来ています(キー-バリューストアのlookupコマンドの生成を考えてください。実際に値に関連付けられているキーを検索する呼び出しを生成できる可能性が欲しいでしょう)。自動生成されたコマンドジェネレーターは少し単純にならざるを得ません。この単純さを軽減するために、ユーザーが以下を提供できるようにしています:

  • 特定のコマンド生成に適用される重み。例えば、キューを所有していないドメインでのwork stealing queueのpushコマンドの生成を無効にするのに便利です。
  • モデルの現在の状態に対してコマンドジェネレーターがパラメータ化されていることを活用したい場合の、完全なコマンドジェネレーター実装。

Happy testing!

OCaml 5.5.0の3番目のアルファリリース

アーカイブ: https://discuss.ocaml.org/t/third-alpha-release-for-ocaml-5-5-0/17928/1

octachronによるアナウンス

OCaml 5.5.0の進行中の安定化作業の進展に伴い、OCaml 5.5.0の3番目のアルファリリースを発表できることを嬉しく思います。

2番目のアルファが未リリースだったのは、再配置可能なコンパイラとブートストラッピング間の予期しない相互作用によるものです。この問題はこの3番目のアルファで修正されています。

この重要な変更以外に、最初のアルファと比較して、この新しいアルファリリースには以下が含まれています

  • 2つのコード生成の修正
  • 3つの型システムの修正
  • 1つの標準ライブラリの修正

(完全なリストについては下記のChangelogを参照してください)。

全体として、5.5.0はかなり安定化しているようで、ほとんどの開発ツールのアルファバージョンが利用可能です。したがって、4月初めにベータリリースに切り替える予定です。

より具体的には、エコシステムの安定化の進行状況は、opamの5.5.0 readiness meta-issueで追跡されています。

このアルファリリース用のコンパニオンアルファopamリポジトリも利用可能です

$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git

これには、OCaml 5.5.0向けに更新中のopamパッケージの進行中のアルファリリースが含まれています。

最終リリースは依然として5月から6月の間を予定しています。

バグを見つけた場合は、issue trackerに報告してください。

新機能とバグ修正の完全なリストに興味がある場合は、更新されたOCaml 5.5.0のchangelogが利用可能です。

Happy hacking, Florian Angeletti for the OCaml team

CS6868 Concurrent Programming – コース教材

アーカイブ: https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1

KC Sivaramakrishnanによるアナウンス

私は現在、IIT Madrasで「CS6868: Concurrent Programming」というコースを教えています。このコースの目標は、OCaml 5を使用して並行および並列プログラミングを教え、その後、安全で高速な並列性のためにOxCamlを紹介することです。OCaml 5の部分はカバーし終え、次にOxCamlを始める予定です。コミュニティとコース教材を共有するのが良いと思いました。

https://github.com/kayceesrk/cs6868_s26

CS6868はOCamlの知識を前提としており、私はCS3100: Paradigms of Programmingでそれをカバーしています。このコースには、すべての講義、ノートブック、課題を含むYouTubeプレイリストがあります。

CS6868は並列性の基礎を構築し、線形化可能性による正当性の推論、スピンロック、ミューテックス、条件変数による並列プログラミング、メモリ一貫性モデル、パフォーマンスの基礎、ノンブロッキングデータ構造(ロックフリーなリンクリスト、キュー、スタック)をカバーします。ここでは、コースはThe Art of Multiprocessor Programmingという本に密接に従っています。すべてのコードはOCaml 5で書き直されています。書籍の内容に加えて、コースはOCamlの緩和メモリモデルもカバーしています。また、qcheck-linとqcheck-stmを使用した並行プロパティベーステストや、TSANを使用したデータ競合検出など、かなり多くのOCaml固有のツールもカバーしています。

その後、コースはControl structures in programming languages: from goto to algebraic effectsの特定の部分に従って、エフェクトハンドラーの基礎をカバーします。次に、軽量スレッド、バッファ付きチャネル、選択的通信(Concurrent MLの精神で)、ネストされた並列プログラミング(domainslibのように)、基本的な非同期IO(Eioのように)を備えた、Go風のマルチコア対応並行プログラミングライブラリを完全に構築します。

コース教材を一通り学習すれば実行可能な、増え続けるコースプロジェクトのコレクションがあります。コースプロジェクトのアイデアがあればお知らせください。

このコースは、これまで行ってきた多くのチュートリアルからの内容をまとめようとしています。フィードバックがあればお知らせください。小さな問題を修正するPRは受け入れますが、これは進行中のコースであるため、大きなPRはおそらく受け入れられません。これを作るのを楽しんだのと同じくらい、これに取り組むのを楽しんでいただけることを願っています。
