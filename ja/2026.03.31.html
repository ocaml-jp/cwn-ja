MerlinのSignature-help機能の改善

Archive: https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1

Charlène_Gros が発表しました

Taridesの私たちは、Merlinのsignature-help機能に加えられたいくつかの改善をお知らせできることを大変嬉しく思います！

念のため説明しますと、signature-helpはMerlinの機能で、関数のシグネチャを表示してアクティブなパラメータをハイライトするものです。エディタでは通常、ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。たとえば、以下の画像はVSCodeにおける関数exec_progのsignature-helpを示しています。

signature-helpコマンドの実装にはいくつかの問題点があり、それらが修正されました。これらの修正により、機能がより使いやすくなります。一部の改善はすでに最新のMerlinおよびocaml-lspのリリースに含まれており、残りは次のリリースに含まれる予定です。

改善内容：

  • Merlinは関数のパラメータ上でのみsignature-help情報を表示するようになり、関数名では表示されなくなりました。
  • signature-helpはアクティブなパラメータに対してのみ関数情報を表示するようになり、最後のパラメータの後に最初のパラメータに戻ってハイライトするループがなくなりました。
  • let .. inバインディングの内部で使用された場合、関数呼び出しの後にinがまだ書かれていなくてもsignature-helpがトリガーされるようになりました。
  • アクティブなパラメータがオプショナルパラメータの書き込みを開始すると、オプショナルパラメータが検出されてハイライトされるようになりました。

ortac-0.8 Domainsを使った仕様駆動テスト

Archive: https://discuss.ocaml.org/t/ann-ortac-0-8-specification-driven-testing-with-domains/17927/1

Nicolas Osborne が発表しました

こんにちは！Taridesの私たちは、仕様駆動テストのためのortac-0.8.0のリリースを発表できることをとても興奮しています！

ortacはGospelエコシステムのツールです。核心的なアイデアは、Gospelの仕様言語のサブセットをOCamlコードに変換し、これらの変換を使ってランタイムチェックを生成することです。

ortacはプラグインアーキテクチャを採用しています。opam経由でインストールできます： 

$ opam install ortac-wrapper    # install the wrapper plugin for runtime assertion checking, with all the other necessary parts (runtime and cli)
$ opam install ortac-qcheck-stm # install the qcheck-stm plugin that generates QCheck-STM tests, with all the other necessary parts (runtime and cli)
$ opam install ortac-dune       # install the dune plugin to generate dune boilerplate to use the two previous plugins

このリリースは、Ortac/QCheck-STMがQCheck-STMテストフレームワークのより多くの機能、すなわち並列コンテキストでのテストとコマンド生成の柔軟性を活用できるようにすることに焦点を当てています。

動作の仕組みについてはドキュメントをご覧ください。

並列コンテキストでのテストについては、バージョン0.2でのバグレポート機能の導入とバージョン0.4でのSUT返却関数のカバレッジにより、並列コンテキストでのQCheck-STMテストの生成が（手書きのQCheck-STMテストでは非常に簡単であるにもかかわらず）容易ではなくなっていました。これが修正されました！

バグレポートの情報は、実際のテスト（実際の実行結果とモデルからの結果を比較する）を行う関数で部分的に収集されます。つまり、並列テストにバグレポート機能を実装するには、QCheck-STM+Domainsの核心であり、テストを実行する際に信頼している関数（いわばTrusted Testing Base）を書き直す必要があります。同じ動作が維持されること、生成されたテストが対応する手書きのQCheck-STMテストと同じセマンティクスを持つことを確認するために、ロジックの保存が明らかになるよう情報収集を導入したコミットのdiffを最小化しました（最終的には、人間によるレビューが信頼の根拠です）。

バージョン0.4以降、SUT返却関数はテストに含まれています。新しく作成されたSUTは、次の呼び出しの引数として選択されるSUTのストアに追加されます。ここでの問題は、並列コンテキストでこれらをどう扱うかです：並列ブランチで作成されたSUTをグローバルストアに追加することはできません。ドメイン間で共有されることを想定していないからです。シンプルな設計を選択し、並列コンテキストに入ったら新しく作成されたSUTのストアへの保存を停止します。これらのSUT返却関数は、テストのシーケンシャル部分（シーケンシャルモードおよび/または並列モードのシーケンシャルプレフィックス）で引き続き完全にテストされますのでご安心ください。

QCheck-STMテストフレームワークの強みの一つは、コマンドジェネレーターの柔軟性です。この柔軟性はQCheck.Gen~APIそのものと、QCheck-STMコマンドジェネレーターが現在のモデルの状態にパラメータ化されているという事実から来ています（キーバリューストアの~lookupコマンドの生成を考えてみてください。実際に値が関連付けられているキーを検索する呼び出しを生成するチャンスを得たいはずです）。自動生成されたコマンドジェネレーターはやや素朴になりがちです。この素朴さを緩和するために、ユーザーが以下を提供できるようにしています：

  • 特定のコマンド生成に適用するウェイト。例えば、work stealing queueでキューを所有していないドメインでのpushコマンドの生成を無効化するのに有用です。
  • コマンドジェネレーターがモデルの現在の状態にパラメータ化されていることを活用したい場合の完全なコマンドジェネレーター実装。

ハッピーテスティング！

OCaml 5.5.0 の第3アルファリリース

Archive: https://discuss.ocaml.org/t/third-alpha-release-for-ocaml-5-5-0/17928/1

octachron が発表しました

OCaml 5.5.0の安定化作業の進展に伴い、OCaml 5.5.0の第3アルファリリースを発表できることを嬉しく思います。

欠番となった第2アルファは、再配置可能なコンパイラとブートストラップの間の予期せぬ相互作用により未リリースとなりました。この問題は第3アルファで修正されています。

この重要な変更に加えて、第1アルファとの比較でこの新しいアルファリリースには以下が含まれています：

  • コード生成のバグ修正が2件
  • 型システムのバグ修正が3件
  • 標準ライブラリのバグ修正が1件

（完全なリストについては以下のChangelogをご覧ください。）

全体的に、5.5.0の安定化は順調に進んでいるようで、ほとんどの開発ツールのアルファバージョンが利用可能です。そのため、4月初旬にベータリリースに切り替える予定です。

より詳しくは、エコシステムの安定化の進捗状況はopamの5.5.0 readiness meta-issueで追跡されています。

このアルファリリース用のコンパニオンアルファopamリポジトリも利用可能です： 

$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git

これにはOCaml 5.5.0向けに更新中のopamパッケージの進行中のアルファリリースが含まれています。

最終リリースは引き続き5月から6月の間を予定しています。

バグを見つけた場合は、issue trackerにご報告ください。

新機能とバグ修正の完全なリストに興味がある方は、更新されたOCaml 5.5.0のchangelogをご覧ください。

Happy hacking, Florian Angeletti（OCamlチームを代表して）

第1アルファからの変更点

  • コンパイラ成果物の修正
    • ブートストラップとの互換性が向上した再配置可能なコンパイラ
    • js_of_ocamlが報告したcompiler-libs関連の問題を修正するためのブートストラップ
  • コード生成のバグ修正
    • #14583: 特定の状況でコンパイルの誤りを引き起こす可能性があった線形スキャンスピルヒューリスティックのバグを修正。 (Nicolás Ojeda Bär、Vincent Lavironによるレビュー)
    • #13693, #14514: s390x: PLTの遅延バインディングトランポリンがFPRをOCamlのファイバースタックに保存することで引き起こされるlibasmrun_shared.soによるヒープ破損を修正。s390xコードエミッターでPLT呼び出しをGOT間接呼び出しに置き換え。 (Zane Hambly、David AllsoppとXavier Leroyによるレビュー)
  • 型システムのバグ修正
    • #14434, #14652: check_counter_example_patを多相型から保護し、型の健全性を回復。 (Stephen DolanとJacques Garrigue、Alistair O'Brienによる報告とレビュー)
    • #14603, #14604: 多相型とアンボックスドコンストラクタの混合時のCtype.applyの失敗を回避。 (Gabriel SchererとStefan Muenzel、Brandon Strideによる報告、 Florian Angelettиによるレビュー)
    • #14626, #14675: オプショナル引数が消去不可能かどうかを決定する際に、モジュール依存関数を考慮するように。 (Alistair O'BrienとFlorian Angeletti、Gabriel Schererによるレビュー)
  • 標準ライブラリの修正
    • #14635: caml_floatarray_gatherのバグを修正。これによりFloat.Array.sub、Float.Array.append、Float.Array.concat（空の場合）の結果が[||]と等しくならない問題を解決。 (Marc Lasson、Gabriel Schererによるレビュー)
  • ドキュメントの更新
    • #13590: Windows上でのランタイムによる*および?ワイルドカードのコマンドライン自動展開を文書化。 (Benjamin Sigonneau、Nicolás Ojeda Bärによるレビュー)

CS6868 並行プログラミング — コース資料

Archive: https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1

KC Sivaramakrishnan が発表しました

現在、IIT Madrasで「CS6868: Concurrent Programming（並行プログラミング）」というコースを教えています。このコースの目標は、OCaml 5を使って並行・並列プログラミングを教え、その後、安全で高速な並列処理のためにOxCamlを紹介することです。OCaml 5の部分は済みましたので、次はOxCamlを始める予定です。コース資料をコミュニティと共有するのが良いと思いました。

https://github.com/kayceesrk/cs6868_s26

CS6868はOCamlの知識を前提としており、それはCS3100: Paradigms of Programmingでカバーしています。このコースには全講義のノートブックと課題を含むYouTubeプレイリストがあります。

CS6868は並列性の基礎、線形化可能性による正確性の推論、スピンロック・ミューテックス・条件変数・メモリ一貫性モデルによる並列プログラミング、パフォーマンスの基礎とノンブロッキングデータ構造（ロックフリー連結リスト、キュー、スタック）を扱います。ここでは、コースはThe Art of Multiprocessor Programmingという本に密接に従っています。すべてのコードはOCaml 5で書き直されています。本のコンテンツに加え、コースではOCamlの緩和メモリモデルもカバーしています。また、qcheck-linとqcheck-stmを使った並行プロパティベーステストや、TSANを使ったデータレース検出など、OCaml固有のツールも相当量カバーしています。

その後、Control structures in programming languages: from goto to algebraic effectsの特定の部分に従って、エフェクトハンドラーの基礎をカバーします。続いて、軽量スレッド・バッファードチャンネル・選択的通信（Concurrent MLの流れで）・ネスト並列プログラミング（domainslibのように）・基本的な非同期IO（Eioのように）を備えた本格的なGoのようなマルチコア対応並行プログラミングライブラリを構築します。

コースプロジェクトのコレクションも増えており、コース資料を通読すれば実施可能なものです。コースプロジェクトのアイデアがあればお知らせください。

このコースは、過去に行った多くのチュートリアルからのコンテンツをまとめようとしています。フィードバックがあればお知らせください。小さな問題を修正するPRは受け付けますが、進行中のコースであるため、大きなPRは受け付けられない可能性が高いです。作成するのと同じくらい楽しんで取り組んでいただければ幸いです。
