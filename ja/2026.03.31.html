Merlin の signature-help 機能の改善

Archive: https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1

Charlène_Gros さんのアナウンス

Tarides のチームは、Merlin の signature-help 機能に加えられた改善をお知らせできることを大変嬉しく思います！

おさらいとして、~signature-help~ は Merlin の機能で、関数のシグネチャを表示し、アクティブなパラメータをハイライトします。エディタでは通常、ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。たとえば、以下の画像は VSCode での関数 exec_prog に対する signature-help の表示例です。

signature-help コマンドの実装にはいくつかの問題点がありましたが、それらが修正されました。これらの修正により、機能がより快適に使えるようになりました。改善の一部はすでに最新の Merlin および ocaml-lsp のリリースに含まれており、残りは次のリリースに含まれる予定です。

改善内容：

  • Merlin は関数のパラメータ上でのみ signature-help の情報を表示するようになり、関数名の上では表示されなくなりました。
  • signature-help はアクティブなパラメータの情報のみを表示するようになり、最後のパラメータの後に最初のパラメータに戻ってハイライトするループがなくなりました。
  • 関数呼び出しの後に in がまだ書かれていない let .. in バインディングの内部でも signature-help がトリガーされるようになりました。
  • オプションパラメータの記述を始めた際に、そのパラメータが検出されてハイライトされるようになりました。

ortac-0.8 - Domains を用いた仕様駆動テスト

Archive: https://discuss.ocaml.org/t/ann-ortac-0-8-specification-driven-testing-with-domains/17927/1

Nicolas Osborne さんのアナウンス

こんにちは！Tarides のチームは、仕様駆動テストのための ortac-0.8.0 のリリースをお知らせできることを大変嬉しく思います！

ortac は Gospel エコシステムのツールです。中心的なアイデアは、Gospel 仕様言語のサブセットを OCaml コードに変換し、その変換を使ってランタイムチェックを生成することです。

ortac はプラグインアーキテクチャを採用しています。opam でインストールできます： 

$ opam install ortac-wrapper    # ランタイムアサーションチェック用のラッパープラグインをインストール（ランタイムと cli など必要な部分を含む）
$ opam install ortac-qcheck-stm # QCheck-STM テストを生成する qcheck-stm プラグインをインストール（ランタイムと cli など必要な部分を含む）
$ opam install ortac-dune       # 前述の 2 つのプラグインを使用するための dune ボイラープレートを生成する dune プラグインをインストール

このリリースは、QCheck-STM テストフレームワークのより多くの機能を Ortac/QCheck-STM が活用できるようにすることに焦点を当てています。具体的には、並列コンテキストでのテストとコマンド生成の柔軟性です。

動作の仕組みについては ドキュメント をご覧ください。

並列コンテキストでのテストについて、バージョン 0.2 でのバグレポート機能の導入とバージョン 0.4 での SUT を返す関数のカバレッジにより、並列コンテキストでの QCheck-STM テストの生成が容易ではなくなっていました（手書きの QCheck-STM テストでは比較的簡単です）。この問題が修正されました！

バグレポートの情報は、実際のテスト（実際の実行結果とモデルの結果を比較する）を行う関数内で一部収集されます。つまり、並列テストのためのバグレポート機能を実装するには、この関数を書き直す必要があります。この関数は QCheck-STM+Domains の核心であり、テスト実行時に信頼している部分です（いわば Trusted Testing Base です）。同じ動作が維持されること、生成されたテストが対応する手書きの QCheck-STM テストと同じセマンティクスを持つことを確認するために、情報収集を導入したコミットの差分を最小化し、ロジックの保全が明確になるようにしました（最終的には、人間によるレビューが信頼の根拠です）。

バージョン 0.4 以降、SUT を返す関数がテストに含まれています。新しく作成された SUT は、次の呼び出しの引数として選択される SUT のストアに追加されます。ここで問題となるのは、並列コンテキストでどう扱うかです：並列ブランチで作成された SUT をグローバルなストアに追加することはできません。ドメイン間で共有されることを想定していないからです。シンプルな設計として、並列コンテキストに入ったら新しく作成された SUT のストアへの追加を停止するようにしました。ご安心ください。これらの SUT を返す関数は引き続き、テストのシーケンシャル部分（シーケンシャルモードおよび並列モードのシーケンシャルプレフィックス）で完全にテストされます。

QCheck-STM テストフレームワークの強みの一つは、コマンドジェネレータの柔軟性です。この柔軟性は QCheck.Gen API 自体と、QCheck-STM コマンドジェネレータが現在のモデルの状態にパラメータ化されている事実から来ています（例えばキーバリューストアの lookup コマンドの生成を考えると、実際に値が関連付けられているキーを検索する呼び出しを生成できる確率が欲しいですよね）。自動生成されたコマンドジェネレータはやや素朴になりがちです。この素朴さを軽減するために、ユーザーが以下を提供できるようにしました：

  • 特定のコマンド生成に適用するウェイト。例えば、キューを所有していないドメインでのワークスティールキューへの push コマンドの生成を無効にするのに便利です。
  • ユーザーがコマンドジェネレータを現在のモデルの状態にパラメータ化された形で活用したい場合の、完全なコマンドジェネレータの実装。

Happy testing!

OCaml 5.5.0 の第3アルファリリース

Archive: https://discuss.ocaml.org/t/third-alpha-release-for-ocaml-5-5-0/17928/1

octachron さんのアナウンス

OCaml 5.5.0 の安定化作業の進捗に伴い、OCaml 5.5.0 の第3アルファリリースをお知らせできることを嬉しく思います。

第2アルファがリリースされなかったのは、再配置可能コンパイラとブートストラッピングの間に予期しない相互作用が発生したためです。この問題は今回の第3アルファで修正されています。

この重要な変更に加え、第1アルファと比較してこの新しいアルファリリースには以下が含まれています：

  • コード生成のバグ修正 2件
  • 型システムのバグ修正 3件
  • 標準ライブラリの修正 1件

（詳細は以下の Changelog をご覧ください。）

全体として、5.5.0 の安定化は順調に進んでおり、ほとんどの開発ツールのアルファ版も利用可能になっています。そのため、4月初旬にベータリリースに移行する予定です。

エコシステムの安定化の進捗状況は opam 上の 5.5.0 readiness メタイシューで追跡されています。

このアルファリリースと合わせて、コンパニオンのアルファ opam リポジトリも利用可能です： 

$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git

このリポジトリには、OCaml 5.5.0 向けに更新中の opam パッケージのアルファリリース（進行中のもの）が含まれています。

最終リリースは引き続き 5月から6月の間を予定しています。

バグを発見した場合は、issue tracker でご報告ください。

新機能とバグ修正の完全なリストにご興味のある方は、更新された OCaml 5.5.0 の Changelog をご覧ください。

Happy hacking, Florian Angeletti（OCaml チームを代表して）

第1アルファからの変更点

  • コンパイラ成果物の修正
    • ブートストラップに対してより親和的な再配置可能コンパイラ
    • js_of_ocaml が報告した compiler-libs 関連の問題を修正するためのブートストラップ
  • コード生成のバグ修正
    • #14583：特定の状況でコンパイル誤りを引き起こす可能性があった線形スキャンスピルヒューリスティックのバグを修正。 (Nicolás Ojeda Bär、レビュー：Vincent Laviron)
    • #13693、#14514：s390x：PLT 遅延バインディングトランポリンが FPR を OCaml のファイバースタックに保存することによって引き起こされる libasmrun_shared.so でのヒープ破壊を修正。s390x コードエミッターで PLT 呼び出しを GOT 間接呼び出しに置き換え。 (Zane Hambly、レビュー：David Allsopp および Xavier Leroy)
  • 型システムのバグ修正
    • #14434、#14652：多相型に対して check_counter_example_pat を保護し、型の健全性を回復。 (Stephen Dolan および Jacques Garrigue、レポートおよびレビュー：Alistair O'Brien)
    • #14603、#14604：多相型とアンボックスドコンストラクタを混在させた場合の Ctype.apply の失敗を回避。 (Gabriel Scherer および Stefan Muenzel、レポート：Brandon Stride、レビュー：Florian Angeletti)
    • #14626、#14675：オプション引数が消去不可かどうかを判断する際に、モジュール依存関数を考慮するよう修正。 (Alistair O'Brien および Florian Angeletti、レビュー：Gabriel Scherer)
  • 標準ライブラリの修正
    • #14635：~Float.Array.sub~、~Float.Array.append~、~Float.Array.concat~（空の場合）の結果が [||] と等しくならない原因となっていた caml_floatarray_gather のバグを修正。 (Marc Lasson、レビュー：Gabriel Scherer)
  • ドキュメントの更新
    • #13590：Windows 上でのランタイムによる * および ? ワイルドカードのコマンドライン自動展開をドキュメント化。 (Benjamin Sigonneau、レビュー：Nicolás Ojeda Bär)

CS6868 並行プログラミング – コース教材

Archive: https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1

KC Sivaramakrishnan さんのアナウンス

現在、IIT Madras で「CS6868：Concurrent Programming（並行プログラミング）」というコースを教えています。このコースの目標は、OCaml 5 を使った並行・並列プログラミングを教え、その後、安全で高速な並列処理のために OxCaml を紹介することです。OCaml 5 の部分はカバーし終わり、次は OxCaml を始める予定です。コース教材をコミュニティと共有するのが良いと思いました。

https://github.com/kayceesrk/cs6868_s26

CS6868 は OCaml の知識を前提としており、それは CS3100：Paradigms of Programming でカバーしています。このコースには YouTube プレイリストがあり、すべての講義、ノートブック、課題が含まれています。

CS6868 では、並列性の基礎、線形化可能性による正確性の推論、スピンロック・ミューテックス・条件変数による並列プログラミング、メモリ整合性モデル、パフォーマンスの基礎と非ブロッキングデータ構造（ロックフリーリンクリスト、キュー、スタック）を構築します。ここでは、コースは書籍The Art of Multiprocessor Programming に沿って進めています。すべてのコードは OCaml 5 で書き直しています。書籍の内容に加え、OCaml の緩和メモリモデルもカバーしています。また、qcheck-lin および qcheck-stm を使った並行プロパティベーステストや TSAN を使ったデータレース検出 など、OCaml 固有のツールも多く取り上げています。

続いて、Control structures in programming languages: from goto to algebraic effects の特定の部分に沿って、エフェクトハンドラーの基礎をカバーします。その後、軽量スレッド、バッファードチャネル、選択的通信（Concurrent ML の流れで）、ネストされた並列プログラミング（domainslib のように）、基本的な非同期 IO（Eio のように）を備えた、Go 風のフルスペックのマルチコア対応並行プログラミングライブラリを構築します。

コースプロジェクトも増えています。コース教材を一通り学べば取り組めるはずです。コースプロジェクトのアイデアがあればお知らせください。

このコースは、過去に行った 多くの チュートリアル の内容をまとめようとしています。フィードバックがあればお知らせください。小さな問題を修正する PR は受け入れますが、現在進行中のコースであるため、大きな PR はおそらく受け入れられません。皆さんがこれを楽しんで取り組んでいただけることを、私が作る楽しさと同じくらい願っています。
