# Merlinのsignature-help機能の改善 {#1}

アーカイブ:
<https://discuss.ocaml.org/t/ann-improve-signature-help-feature-in-merlin/17920/1>

## Charlène_Grosによるアナウンス

Taridesでは、Merlinの`signature-help`機能に対する改善を発表できることを大変嬉しく思います!

念のため、`signature-help`はMerlinの機能で、関数のシグネチャを表示し、アクティブなパラメータをハイライトします。エディタでは、これは通常ツールチップ、フローティングウィンドウ、またはミニバッファに表示されます。例えば、次の画像はVSCodeでの関数`exec_prog`の`signature-help`を表示しています。

```{=org}
#+attr_html: :width 80%
```
<https://us1.discourse-cdn.com/flex020/uploads/ocaml/original/2X/4/46e99c5c69bab991b8dd6a6a7494db5c2f32e07b.png>

signature-helpコマンドの実装にはいくつかの欠点がありましたが、修正されました。これらの修正により、この機能の使用がはるかに快適になりました。これらの改善の一部は、最新のMerlinとocaml-lspのリリースにすでに含まれており、その他は次のリリースに含まれる予定です。

結果:

- Merlinは関数パラメータに対してのみ`signature-help`情報を表示し、関数名には表示しなくなりました。
- `signature-help`はアクティブなパラメータに対してのみ関数の情報を表示し、最後のパラメータの後に最初のパラメータをハイライトするループバックをしなくなりました。
- 関数呼び出しの後に`in`がまだ書かれていない`let .. in`バインディング内で使用されても、`signature-help`がトリガーされるようになりました。
- アクティブなパラメータがオプションパラメータの記述を開始したときに、オプションパラメータが検出されてハイライトされるようになりました。

# ortac-0.8 Domainsを用いた仕様駆動テスト {#4}

アーカイブ:
<https://discuss.ocaml.org/t/ann-ortac-0.8-specification-driven-testing-with-domains/17927/1>

## Nicolas Osborneによるアナウンス

こんにちは! Taridesでは、仕様駆動テストのためのortac-0.8.0のリリースを発表できることを大変嬉しく思います!

`ortac`は[Gospel](https://github.com/ocaml-gospel)エコシステムのツールです。中核となるアイデアは、Gospel仕様言語のサブセットをOCamlコードに変換し、これらの変換を使用してランタイムチェックを生成することです。

`ortac`はプラグインアーキテクチャを採用しています。opam経由でインストールできます:

``` example
$ opam install ortac-wrapper    # ランタイムアサーションチェック用のwrapperプラグインを、必要な他のすべての部分(runtimeとcli)とともにインストール
$ opam install ortac-qcheck-stm # QCheck-STMテストを生成するqcheck-stmプラグインを、必要な他のすべての部分(runtimeとcli)とともにインストール
$ opam install ortac-dune       # 前述の2つのプラグインを使用するためのduneボイラープレートを生成するduneプラグインをインストール
```

このリリースは、OrtacとQCheck-STMがQCheck-STMテストフレームワークのより多くの機能を活用できるようにすることに焦点を当てています。具体的には、並列コンテキストでのテストとコマンド生成の柔軟性です。

動作の仕組みについては[ドキュメント](https://ocaml-gospel.github.io/ortac/ortac-qcheck-stm/index.html)をご覧ください。

並列コンテキストでのテストに関しては、バージョン0.2でのバグレポート機能の導入と、バージョン0.4でのSUT返却関数のカバレッジにより、並列コンテキストでのQCheck-STMテストの生成が容易ではなくなりました(手書きのQCheck-STMテストでは非常に簡単です)。これが修正されました!

バグレポートの情報は、実際のテストを行う関数(実際の実行結果とモデルの結果を比較)で部分的に収集されます。これは、並列テストのバグレポート機能を実装するには、QCheck-STM+Domainsの核心であり、テスト実行時に信頼している部分(Trusted Testing Baseとでも言いましょうか)であるこの関数を書き直す必要があることを意味します。同じ動作を維持していること、生成されたテストが対応する手書きのQCheck-STMテストと同じセマンティクスを持つことを確認するために、情報収集を導入する[コミット](https://github.com/ocaml-gospel/ortac/commit/ef8d598ccbd6c9987648e201ebffb6abb5a3d610)の差分を最小化し、ロジックの保存が明らかになるようにしました(結局のところ、人間のレビューこそが信頼するものです)。

バージョン0.4以降、SUT返却関数がテストに含まれています。新しく作成されたSUTは、次の呼び出しの引数として選択されるSUTのストアに追加されます。ここでの疑問は、並列コンテキストでこれらをどうするかです。並列ブランチで作成されたSUTをグローバルストアに追加することはできません。ドメイン間で共有されることは想定されていないためです。私たちは、並列コンテキストに入ったら新しく作成されたSUTの保存を停止するというシンプルな設計を選択しました。心配ありません。これらのSUT返却関数は、テストのシーケンシャル部分(シーケンシャルモードおよび/または並列モードのシーケンシャルプレフィックス)で完全にテストされます。

QCheck-STMテストフレームワークの強力さの1つは、コマンドジェネレータの柔軟性です。この柔軟性は、`QCheck.Gen` API自体と、QCheck-STMコマンドジェネレータが現在のモデルの状態でパラメータ化されているという事実から来ています(キーバリューストアの`lookup`コマンドの生成を考えてみてください。実際に値に関連付けられているキーを検索する呼び出しを生成する機会を持ちたいでしょう)。自動生成されたコマンドジェネレータは、少し素朴にならざるを得ません。この素朴さを軽減するために、ユーザーが以下を提供できるようにしています:

- 特定のコマンド生成に適用される重み。例えば、キューを所有していないはずのドメインで[ワークスティーリングキュー](https://ocaml-multicore.github.io/saturn/saturn/Saturn/Work_stealing_deque/index.html#val-push)の`push`コマンドの生成を無効にするのに便利です。
- ユーザーがモデルの現在の状態でパラメータ化されるコマンドジェネレータを活用したい場合の完全なコマンドジェネレータ実装。

ハッピーテスティング!

# OCaml 5.5.0の第3アルファリリース {#5}

アーカイブ:
<https://discuss.ocaml.org/t/third-alpha-release-for-ocaml-5-5-0/17928/1>

## octachronによるアナウンス

OCaml 5.5.0の進行中の安定化作業の進展により、OCaml 5.5.0の第3アルファリリースを発表できることを嬉しく思います。

第2アルファは、再配置可能コンパイラとブートストラッピング間の予期しない相互作用のためリリースされませんでした。この問題は第3アルファで修正されました。

この重要な変更に加えて、第1アルファと比較して、この新しいアルファリリースには以下が含まれます:

- 2つのコード生成修正
- 3つの型システム修正
- 1つの標準ライブラリ修正

(完全なリストについては、以下のChangelogを参照してください)。

全体として、5.5.0は非常に順調に安定化しており、ほとんどの開発ツールのアルファバージョンが利用可能です。したがって、4月初めにベータリリースに切り替える予定です。

より正確には、エコシステムの安定化に関する進行中の進捗は、[opamの5.5.0対応メタissue](https://github.com/ocaml/opam-repository/issues/29463)で追跡されています。

このアルファリリース用の付属のアルファopamリポジトリも利用可能です

``` example
$ opam repo add alpha git+https://github.com/kit-ty-kate/opam-alpha-repository.git
```

これには、OCaml 5.5.0用に更新中のopamパッケージの進行中のアルファリリースが含まれています。

最終リリースは、引き続き5月から6月の間を予定しています。

バグを発見した場合は、[issue tracker](https://github.com/ocaml/ocaml/issues)で報告してください。

新機能とバグ修正の完全なリストに興味がある場合は、[OCaml 5.5.0の更新されたchangelog](https://github.com/ocaml/ocaml/blob/5.5/Changes)が利用可能です。

ハッピーハッキング、Florian Angeletti for the OCaml team

### 第1アルファからの変更点

1. コンパイラアーティファクト修正

   - よりブートストラップフレンドリーな再配置可能コンパイラ
   - js_of_ocamlから報告されたcompiler-libs関連の問題を修正するブートストラップ

   - [#13693](https://github.com/ocaml/ocaml/issues/13693)、[#14514](https://github.com/ocaml/ocaml/issues/14514): s390x: PLT遅延バインディングトランポリンがFPRをOCamlのファイバスタックに保存することによって引き起こされるlibasmrun_shared.soでのヒープ破損を修正。s390xコードエミッタでPLT呼び出しをGOT間接呼び出しに置き換え。(Zane Hambly、レビュー: David AllsoppとXavier Leroy)

3. 型システムバグ修正

   - [#14434](https://github.com/ocaml/ocaml/issues/14434)、[#14652](https://github.com/ocaml/ocaml/issues/14652): check_counter_example_patを多相型から保護し、型の健全性を回復。(Stephen DolonとJacques Garrigue、報告とレビュー: Alistair O'Brien)

   - [#14603](https://github.com/ocaml/ocaml/issues/14603)、[#14604](https://github.com/ocaml/ocaml/issues/14604): 多相型とアンボックス化コンストラクタを混在させたときのCtype.apply失敗を回避。(Gabriel SchererとStefan Muenzel、報告: Brandon Stride、レビュー: Florian Angeletti)

   - [#14626](https://github.com/ocaml/ocaml/issues/14626)、[#14675](https://github.com/ocaml/ocaml/issues/14675): オプション引数が消去不可能かどうかを判定する際に、モジュール依存関数を考慮に入れる。(Alistair O'BrienとFlorian Angeletti、レビュー: Gabriel Scherer)

4. 標準ライブラリ修正

   - [#14635](https://github.com/ocaml/ocaml/issues/14635): `caml_floatarray_gather`のバグを修正。このバグにより、`Float.Array.sub`、`Float.Array.append`、`Float.Array.concat`(空の場合)の結果が`[||]`と等しくならなかった。(Marc Lasson、レビュー: Gabriel Scherer)

5. ドキュメント更新

   - [#13590](https://github.com/ocaml/ocaml/issues/13590): Windows下でランタイムによる`*`および`?`ワイルドカードの自動コマンドライン展開を文書化。(Benjamin Sigonneau、レビュー: Nicolás Ojeda Bär)

# CS6868 並行プログラミング -- コース教材 {#7}

アーカイブ:
<https://discuss.ocaml.org/t/ann-cs6868-concurrent-programming-course-materials/17932/1>

## KC Sivaramakrishnanによるアナウンス

現在、IIT MadrasでCS6868: Concurrent Programmingというコースを教えています。このコースの目標は、OCaml 5を使用した並行および並列プログラミングを教え、その後、安全で高速な並列性のために[OxCaml](https://oxcaml.org/)を紹介することです。OCaml 5の部分を終え、次はOxCamlを開始する予定です。コース教材をコミュニティと共有するのが良いと思いました。

<https://github.com/kayceesrk/cs6868_s26>

CS6868はOCamlの知識を前提としており、これは[CS3100: Paradigms of Programming](https://github.com/kayceesrk/cs3100_m20)でカバーしています。このコースには、すべての講義、ノートブック、課題を含む[YouTubeプレイリスト](https://www.youtube.com/watch?v=9R8Oim7YU20&list=PLt0HgEXFOHdkE-NTs87s7QjwYwqeihb-D)が含まれています。

CS6868は、並列性の基礎、線形化可能性による正確性の推論、スピンロック、ミューテックス、条件変数による並列プログラミング、メモリ一貫性モデル、パフォーマンスの基礎、ノンブロッキングデータ構造(ロックフリーのリンクリスト、キュー、スタック)を構築します。ここで、コースは書籍[The Art of Multiprocessor Programming](https://shop.elsevier.com/books/the-art-of-multiprocessor-programming/herlihy/978-0-12-415950-1)に密接に従います。すべてのコードはOCaml 5で書き直されています。書籍の内容に加えて、コースではOCaml緩和メモリモデルもカバーしています。また、[qcheck-linとqcheck-stm](https://github.com/ocaml-multicore/multicoretests/)を使用した並行プロパティベーステストや、[TSANを使用したデータ競合検出](https://ocaml.org/manual/5.3/tsan.html)など、OCaml固有のツールもかなりカバーしています。

その後、コースは[Control structures in programming languages: from goto to algebraic effects](https://xavierleroy.org/control-structures/)の特定の部分に従って、エフェクトハンドラの基礎をカバーします。次に、軽量スレッド、バッファ付きチャネル、選択的通信([Concurrent ML](https://en.wikipedia.org/wiki/Concurrent_ML)のような)、ネストされた並列プログラミング([domainslib](https://github.com/ocaml-multicore/domainslib)のような)、基本的な非同期IO([Eio](https://github.com/ocaml-multicore/eio)のような)を備えた、本格的なGo風のマルチコア対応並行プログラミングライブラリを構築します。

コース教材を学習していれば実行可能な、増え続ける[コースプロジェクト](https://github.com/kayceesrk/cs6868_s26/blob/main/project_ideas.md)のコレクションがあります。コースプロジェクトのアイデアがあればお知らせください。

このコースは、これまでに行った[多くの](https://github.com/ocaml-multicore/ocaml-effects-tutorial)[チュートリアル](https://github.com/ocaml-multicore/ocaml5-parallelism-tutorial)の[内容を](https://github.com/ocaml-multicore/parallel-programming-in-multicore-ocaml)[まとめよう](https://github.com/ocaml-multicore/effects-examples)とするものです。フィードバックがあればお知らせください。小さな問題を修正するPRは受け入れますが、進行中のコースであるため、大きなPRはおそらく受け入れられません。私が作成を楽しんだのと同じくらい、これに取り組むことを楽しんでいただければ幸いです。

# libinput - Linux入力デバイス用OCamlバインディング {#8}

アーカイブ:
<https://discuss.ocaml.org/t/ann-libinput-ocaml-bindings-for-linux-input-devices/17933/1>

## Thomas Leonardによるアナウンス

[libinput-ocaml](https://github.com/talex5/libinput-ocaml)(libinput CライブラリのOCamlバインディング)の最初のリリースを発表できることを嬉しく思います。

libinputは、Waylandコンポジタなどのアプリケーションがマウス、キーボード、タッチパッドなどにアクセスするために使用されます。

[Linux入力デバイス(libinput-ocamlを使用)](https://roscidus.com/blog/blog/2026/03/28/input-devices/)では、Linux上で入力デバイスがどのように機能するかを説明し、このライブラリを使用して小さなゲームを作成する方法を示しています:

